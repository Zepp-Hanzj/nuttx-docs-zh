==========================
在 board.h 中包含文件
==========================

.. warning:: 
    迁移自: https://cwiki.apache.org/confluence/display/NUTTX/Including+Files+in+board.h

board.h 的全局作用域
=======================

``boards/`` 目录下的每个板都必须在板的 ``include/`` 子目录中提供一个名为 ``board.h`` 的头文件。

当板被配置时，将创建指向该板 ``board.h`` 头文件的符号链接，位于 ``include/arch/board/board.h``。该位置的头文件具有 `全局作用域`，意味着它们可以被系统中任何位置的 C/C++ 逻辑包含。例如，``board.h`` 文件可以被任何文件通过以下方式包含：

.. code-block:: C

    #include <arch/board/board.h>

架构特定头文件的受限作用域
======================================================

每个架构还提供内部架构特定头文件，只能被 (1) 架构逻辑本身和 (2) 基于该架构的板逻辑包含。这是通过 NuttX 构建系统仅为架构和板逻辑提供的 CFLAGS 中的特殊包含路径选项实现的。

因此，架构特定头文件的作用域是 `受限的`。任何非架构或板特定逻辑尝试包含架构特定头文件都将导致编译错误。这是预期的行为；这样做是为了防止在整个操作系统中使用此类 `受限的`、不可移植的文件。这样做会降低操作系统的模块化性，并在通往 `意大利面条代码` 的道路上迈出许多巨大的步伐。


由于头文件混合作用域导致的错误
=========================================

``board.h`` 头文件在许多上下文中被包含，包括架构和板特定逻辑以及其他通用 OS 逻辑。例如：

* 架构特定代码可能包含 ``board.h`` 头文件以获取 GPIO 引脚定义或获取配置处理器时钟的配置信息。
* 通用 LED 和按钮驱动程序分别包含板的 ``board.h`` 头文件以获取 LED 和按钮定义。
* 应用程序可能包含 ``board.h`` 头文件以获取例如 ``boardctl()`` 板特定 IOCTL 命令。

在第一个示例中，``board.h`` 头文件通常使用需要从架构特定头文件定义的预处理器符号。例如，GPIO 引脚定义或时钟配置寄存器设置。但是，`您必须避免在 board.h 头文件中包含任何这些架构特定头文件的诱惑。这是被禁止的！` 这将是一个等待在未来导致编译失败的"定时炸弹"。

那么如何避免这种情况？
=======================

避免此头文件包含陷阱的唯一方法是：

* 切勿在 ``board.h`` 头文件中包含架构特定头文件。相反，
* 在包含 ``board.h`` 的 C 文件中包含 ``board.h`` 头文件所需的头文件
* 确保 ``board.h`` 是最后包含的头文件。
