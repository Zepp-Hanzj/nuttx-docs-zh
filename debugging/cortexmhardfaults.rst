分析寄存器转储
===========================

首先，在寄存器转储中：

.. code-block:: console

  R0: ffffffff 00000000 00000016 00000000 00000000 00000000 00000000 00000000
  R8: 100036d8 00000000 00000000 004c4b40 10001370 10005e50 1b02b20b 1b02d596
  xPSR: 41000000 BASEPRI: 00000000 CONTROL: 00000000

``R15`` 是崩溃时的 PC（``1b02d596``）。为了查看它在哪里，我这样做：

.. code-block:: console

  arm-none-eabi-objdump -d nuttx | vi -

当然，你可以使用任何你喜欢的编辑器。无论如何，这将提供 FLASH 内容的完整汇编语言列表以及完整的符号信息。

**提示：** 不熟悉 ARM 汇编语言？试试 ``objdump --source``（或简写为 ``-S``）选项。这将混合显示 C 代码和汇编语言代码，这样你就能看到汇编语言实现了哪些 C 语句。

一旦你在编辑器中打开了 FLASH 镜像，搜索以找到 ``1b02d596`` 处的指令就很简单了。符号信息将准确显示该地址所在的函数，以及可用于将指令关联到原始 C 源文件中确切代码行的上下文。

你还有所有的寄存器内容，所以很容易看出发生了什么（假设你对 Thumb2 汇编语言和 ARM EABI 有一些基本了解）。但通常不容易看出为什么会发生这种情况。

以下说明适用于找出故障发生的原因。

``R14`` 通常包含出错函数调用者的返回地址。这个返回地址中第一位被设置了，但请忽略它（即使用 ``1b02b20a`` 而不是 ``1b02b20b``）。使用上面的 objdump 命令查看它在哪里。

然而，有时 ``R14`` 不是出错函数的调用者。如果出错函数调用了其他函数，那么 ``R14`` 将被覆盖。但没关系，它也会将返回地址压入栈中，我们可以通过分析栈转储找到它。

分析栈转储
========================

任务栈
--------------

要追溯更早的执行历史，你需要分析栈。这是一个向下增长的栈，所以较早的事件在较高的栈地址；最近发生的事件在较低的栈地址。

分析栈的方法基本相同：

1. 从最高的栈地址（最早的）开始，按时间顺序向前（向较低地址）

2. 找到有意义的地址，

3. 使用 ``arm-none-eabi-objdump`` 确定这些地址在代码中的位置。

有意义的地址具有以下特征：

1. 它位于你架构的 FLASH 中。在你的情况下，这些是以 ``0x1a`` 和 ``0x1b`` 开头的地址。其他架构可能有不同的 FLASH 地址，甚至是 RAM 中的地址。

2. 对于 Cortex-M，有意义的地址都是奇数，即位 0 被设置。这是因为随着代码的执行，返回地址（``R14``）会被压入栈中。所有的返回地址都位于 FLASH 中，并且是奇数。

栈转储中的偶数 FLASH 地址通常是对 FLASH 中 ``.rodata`` 的引用，但有时也很有意义。以下是有意义地址的示例（用方括号标出）：

.. code-block:: console

  sp:     10005e50
  User stack:
    base: 10005ed8
    size: 00000f9c
  10005e40: 00000000  00000000  00000000 [1b02d587] 10004900 00000000 005b8d7f 00000000
  10005e60: 1a030f2e  00000000  00000000  00001388  00000000 00000005 10001994 00000000
  10005e80: 00000000  00000000  00000000 [1b02c359] 00000000 00000000 00000000 004c4b40
  10005ea0: 000002ff  00000000  00000000 [1a030f2f] 00000000 00000000 00000000 00000000
  10005ec0: 00000000 [1a030f41] 00000000 [1b02c2a5] 00000000 00000000 ffffffff 00bdeb39

这将给出直到故障发生点的完整回溯。

中断栈
-------------------

注意在某些情况下会列出两个栈。如果（1）启用了中断栈，且（2）故障发生时你正在中断处理程序中，则会出现中断栈：

.. code-block:: console

  Assertion failed at file:armv7-m/up_hardfault.c line: 184 task: hpwork
  sp:     10001eb4
  IRQ stack:
   base: 10001f00
   size: 000003fc
  10001ea0: [1b02d961] 1b03f07e 10001eb4 10005ed8  1a0312ab   1b03f600   000000b8 [1b02d961]
  10001ec0:  00000010  10001f40 00000003 00000000 [1a03721d] [1a037209] [1b02d93b] 00000000
  10001ee0: [1a0371f5] 00000000 00000000 00000000  00000000   00000000  [1a0314a5] 10005d7c

（有意义的地址再次用方括号标出）。

中断栈有时很有意义，例如当中断是由中断级别的逻辑引起时。在这种情况下，它可能不太有意义，因为故障可能是由正常的任务代码引起的，中断栈可能只显示了中断处理逻辑的正常操作。

完整栈分析
-------------------

我在这里提出的只是浏览栈、找到并解释有意义的地址。有时你需要更多信息，需要更详细地分析栈。这也是可能的，因为栈上的每个字都是由于代码中的显式 push 指令而存在的（通常是 Cortex-M 上的 push 指令或其他 ARM 架构上的 stmdb 指令）。这是一项艰苦的工作，但也可以完成，以提供更详细的"发生了什么？"的答案。

恢复硬故障发生时的状态
=============================================

以下是 Mike Smith 的另一个提示：

.. epigraph::

  "...对于像 NuttX 这样捕获硬故障很困难的系统，
  你可以恢复故障的 PC、LR 和 SP（通过检查异常栈），
  然后将这些值写回到适当的处理器寄存器中
  （根据故障情况调整 PC）。

  "这将使你回到故障发生时的应用程序代码位置。
  一些局部变量将显示为无效值（因为在故障发生时
  它们存在于寄存器中并已被异常处理程序覆盖），
  但栈帧、函数参数等应该都正确显示。"
