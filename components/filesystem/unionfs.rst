=================
联合文件系统
=================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

概述
========

此目录包含 NuttX 联合文件系统。联合文件系统提供了一种机制，可以叠加两个不同的已挂载文件系统，使它们显示为一个。一般来说，它的工作方式如下：

  1) 在某个位置挂载文件系统 1，例如 ``/mnt/file1``
  2) 在某个位置挂载文件系统 2，例如 ``/mnt/file2``
  3) 调用 ``mount()`` 将 ``/mnt/file1`` 和 ``/mnt/file2`` 组合并叠加为新的挂载点，例如 ``/mnt/unionfs``。

``/mnt/file1`` 和 ``/mnt/file2`` 将消失并被单个挂载点 ``/mnt/unionfs`` 替代。``/mnt/file1`` 和 ``/mnt/file2`` 下的先前内容将在 ``/mnt/unionfs`` 下合并显示。文件系统 1 中相同相对路径的文件将具有优先权。如果文件系统 2 中存在相同名称和相同相对位置的另一个文件，它将不可见，因为它将被文件系统 1 中的文件遮挡。

参见 ``include/nutts/unionfs.h`` 获取更多信息。

联合文件系统通过在 NuttX 配置文件中选择 ``CONFIG_FS_UNIONFS`` 选项来启用。

免责声明：此联合文件系统当然受到 UnionFS（http://en.wikipedia.org/wiki/UnionFS）的启发，命名上的相似性是不可避免的。但除此之外，NuttX 联合文件系统在规范、使用、设计或实现方面与 UnionFS 项目没有任何关系。

联合文件系统的用途
==============================

此文件的原始动机是将内置函数文件系统（BINFS）与 Web 服务器结合使用。在这种情况下，内置函数提供 CGI 程序。但 BINFS 文件系统不能保存内容。固定内容需要保存在更标准的文件系统中，如 ROMFS。使用此联合文件系统，您可以将 BINFS 挂载点叠加在 ROMFS 挂载点上，提供一个单一目录，其中包含来自 BINFS 文件系统的可执行文件以及来自 ROMFS 文件系统的 Web 内容。

联合文件系统的另一个可能用途是增强或替换 FLASH 文件系统中的文件。例如，假设您有一个产品，在出厂时附带由板载 FLASH 提供的 ROMFS 文件系统中的内容。后来，您可以使用联合文件系统将 SD 卡中的附加文件叠加在该 ROMFS 文件系统上，也许替换 ROMFS 文件。

另一个用例可能是将只读文件系统（如 ROMFS）与可写文件系统（如 RAM 磁盘）叠加。这应该会提供一个具有某些固定内容的可读/可写文件系统。

前缀
========

可以为联合文件系统组合的每个文件系统提供可选的前缀。例如，假设：

* 文件系统 1 是前缀 == NULL 的 ROMFS 文件系统，
* 文件系统 2 是前缀 == "cgin-bin" 的 BINFS 文件系统，以及
* 联合文件系统挂载在 ``/mnt/www``。

那么 ROMFS 文件系统中的内容将出现在 ``/mnt/www``，BINFS 文件系统的内容将出现在 ``/mnt/www/cgi-gin``。

示例配置
======================

* ``boards/sim/sim/sim/unionfs`` - 这是一个模拟器配置，使用 apps/examples/unionfs 中的联合文件系统测试。该测试叠加了两个小型 ROMFS 文件系统，在目录和文件名中有许多冲突。这是测试联合文件系统的好平台，apps/examples/unionfs 是如何配置联合文件系统的好示例。

* ``boards/arm/lpc17xx_40xx/lincoln60/thttpd-binfs`` - 这是使用 THTTPD Web 服务器的示例。它从联合文件系统提供服务，其中固定内容由 ROMFS 文件系统提供，CGI 内容由 BINFS 文件系统提供。

  您可以在 apps/example/thttpd/ 中查看联合文件系统内容目录是如何配置的。

* ``boards/arm/lpc17xx_40xx/olimex-lpc1766stk/thttpd-binfs`` - 这本质上与 lincoln60 配置相同。但是它不起作用，因为 LPC1766 的 RAM 不足以在此配置中支持 THTTPD 应用程序。

有关这些配置的更多信息，请参阅每个板的文档页面。
