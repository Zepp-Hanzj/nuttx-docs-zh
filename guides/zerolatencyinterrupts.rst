=====================================================================
高性能：零延迟中断、可屏蔽嵌套中断
=====================================================================

通用中断处理
==========================

NuttX 包含一个通用中断处理子系统，使得仅使用 IRQ 编号即可方便地处理中断。为了与此通用中断处理系统集成，平台特定代码需要将所有线程状态收集到一个容器 ``struct xcptcontext`` 中。此容器代表线程的完整状态，可以作为一个*线程单元*来保存、恢复和交换。

虽然这种状态保存有许多有用的优点，但它确实需要处理时间。有人向我报告说，在 STM32F4Discovery 板上，这种状态保存大约需要两微秒。这种增加的中断延迟在某些情况下可能是个问题。

此外，RTOS 中各处需要的临界区会暂时暂停中断处理。这增加了在临界区期间变为挂起状态的中断的延迟。由于这对某些中断实例可能发生而对其他实例不会，中断延迟会随时间变化（经历*抖动*）。像上面讨论的增加的延迟一样，这种抖动在某些情况下可能是个问题。

**术语：** 本指南中讨论的概念并非 NuttX 独有。其他 RTOS 有类似的概念，但使用不同的术语。例如，`Nucleus <https://www.embedded.com/design/operating-systems/4461604/Interrupts-in-the-Nucleus-SE-RTOS>`_ RTOS 使用术语 *Native* 和 *Managed* 中断。

绕过通用中断处理
========================================

大多数现代 MCU（如 ARM Cortex-M 系列）通过*向量表*接收和分派中断。向量表是内存中的一个表。表中的每个条目保存对应不同中断的中断处理程序地址。当中断发生时，硬件获取相应的中断处理程序地址并将控制权交给中断处理程序。

在通用中断处理程序的实现中，这些向量中断没有按照硬件设计者的预期使用。相反，它们用于获取 IRQ 编号，然后将控制权转移到通用的中断处理逻辑。

实现更高性能中断并仍然保留通用中断处理逻辑优点的一种方法是简单地用不同的中断处理程序替换向量表中的中断处理程序地址；一个不向通用中断处理逻辑分派，而是分派到你的自定义代码的处理程序。

通常，向量表在 ROM 中。因此，你可以通过修改 ROM 向量表使特定条目指向你的自定义中断处理程序来硬编码特殊的中断向量。或者，如果架构允许，你可以使用 RAM 中的向量表。然后你可以通过直接写入向量表来自由地附加和分离自定义向量处理程序。ARM Cortex-M 移植在启用 ``CONFIG_ARCH_RAMVECTORS`` 选项时提供支持此模式的接口。

那么缺点是什么？有两个：

* 你的自定义中断处理程序不会将其状态收集到 ``struct xcptcontext`` 容器中。因此，它无法与操作系统通信。你的自定义中断处理程序已被"排除在外"，无法再与系统协作。

* 如果你的自定义中断真的是*高性能*的，那么你还必须支持嵌套中断！自定义中断必须具有高优先级，并且必须能够中断通用中断处理逻辑。否则，当你的自定义中断与其他较低优先级的中断发生冲突时，它会偶尔被延迟。

重新加入游戏
==========================

如前所述，自定义中断处理程序无法使用大多数 OS 服务，因为它没有创建 ``struct xcptcontext`` 容器。因此，当它需要与操作系统交互（例如发送信号量、通知线程或发送消息）时，它需要一种机制来"重新加入游戏"。

ARM Cortex-M 系列支持一种使用 *PendSV* 中断的特殊方式：

* 自定义逻辑将使用标准 ``irq_attach()`` 接口与 *PendSV* 中断连接。

* 在自定义中断处理程序中，当需要与 OS 通信时，它将调度 *PendSV* 中断。

* *PendSV* 中断通过通用中断系统分派，因此当附加的 *PendSV* 中断被处理时，它将在可以执行任何必要 OS 交互的上下文中。

对于 ARMv7_M 架构，*PendSV* 中断可以通过以下方式生成：

.. code-block:: c

  up_trigger_irq(NVIC_IRQ_PENDSV);

在其他架构上，可能可以从自定义中断处理程序执行类似软件中断的操作来实现相同的目的。

自定义逻辑将需要在高优先级中断处理程序和 *PendSV* 中断处理程序之间传达感兴趣的事件。对该自定义逻辑的详细讨论超出了本 Wiki 页面的范围。

下表显示了 Cortex-M 系列的优先级级别：

.. code-block::

  IRQ 类型                优先级
  Dataabort              0x00
  High prio IRQ1         0x20	（零延迟中断）
  High prio IRQ2         0x30	（不能在 ISR 中调用 OS API）
  SVC                    0x70
  Disable IRQ            0x80
  (critical-section)
  Low  prio IRQ          0xB0
  PendSV                 0xE0

在此架构上，较低的优先级*数字*意味着更高的优先级。

如你所见，零延迟中断的优先级高于临界区和 SVC，但代价是高优先级 IRQ 不能在 ISR 中调用 OS API。


可屏蔽嵌套中断
==========================

ARM Cortex-M 系列支持一个称为 *BASEPRI* 的功能，可用于禁用低于特定级别的优先级中断。此功能可用于支持可屏蔽嵌套中断。

可屏蔽嵌套中断与零延迟中断的不同之处在于它们遵守系统的中断屏蔽机制。例如，将 BASEPRI 寄存器设置为特定阈值将阻止所有优先级较低或相等的中断。然而，高优先级中断（如不可屏蔽中断或零延迟中断）不受这些屏蔽的影响。

当你有一个需要能够中断系统的高优先级中断，但你也有想要能够屏蔽的较低优先级中断时，这很有用。

下表显示了 Cortex-M 系列的优先级级别：

.. code-block::

  IRQ 类型                 优先级
  Dataabort               0x00
  SVC                     0x70
  Disable IRQ             0x80
  (critical-section)
  High prio IRQ1          0x90    （可屏蔽嵌套中断）
  High prio IRQ2          0xA0    （可以在 ISR 中调用 OS API）
  Low  prio IRQ           0xB0
  PendSV                  0xE0

在此架构上，较低的优先级*数字*意味着更高的优先级。

如你所见，可屏蔽嵌套中断的优先级级别在临界区和低优先级中断之间。在这种情况下，高优先级 IRQ 可以在 ISR 中调用 OS API。


嵌套中断处理
=========================

:doc:`nestedinterrupts` 中提供了一些关于嵌套中断处理的一般说明。在这种情况下，处理嵌套自定义中断更简单，因为不会重新进入通用中断处理程序。相反，通用中断处理程序必须简单地与自定义中断处理程序共存。

可能需要对通用中断处理逻辑进行修改以实现。这里需要说明几点：

* MCU 应支持中断优先级划分，以便自定义中断可以被调度为更高的优先级。

* 通用中断处理程序当前在中断期间禁用中断。相反，它们必须能够在整个中断过程中保持自定义中断启用，同时仍然防止其他标准中断的重入（这可以通过在 Cortex-M 系列中设置中断基优先级级别来实现）。

* 自定义中断处理程序现在可以在任何位置中断通用中断处理程序。逻辑在所有情况下被中断是否安全？有时中断处理程序在操作寄存器时会暂时将 MCU 置于异常状态。确保在任何时候接收中断都是安全的（否则在关键时刻保持中断禁用）。

* 当自定义中断发生时，它是否拥有所有需要的资源？它是否有有效的栈指针？（例如，在 Cortex-M 实现中，当进入自定义中断处理程序时，MSP 可能无效）。

其中一些问题很复杂，因此你应该预期让嵌套中断处理程序工作会有一些复杂性。

Cortex-M3/4 实现
==========================

已为 Cortex-M3/4 系列实现了这样的高优先级嵌套中断处理程序。

以下段落将总结该实现。

配置选项
---------------------

``CONFIG_ARCH_HIPRI_INTERRUPT``

OS 通过将 *BASEPRI* 寄存器设置为 ``NVIC_SYSH_DISABLE_PRIORITY`` 来禁用中断，以便大多数中断不会有执行优先级。*SVCall* 在所有情况下必须具有执行优先级。

在正常情况下，中断不可嵌套，所有中断在 ``NVIC_SYSH_PRIORITY_MIN`` 和 ``NVIC_SYSH_PRIORITY_MAX`` 之间的执行优先级下运行（``NVIC_SYSH_PRIORITY_MAX`` 保留给 *SVCall*）。

如果还定义了 ``CONFIG_ARCH_HIPRI_INTERRUPT``，则支持特殊的高优先级中断。这些不是通常意义上的"嵌套"。这些高优先级中断可以中断正常处理，但在 OS 之外执行（尽管它们可以通过 *PendSV* 中断"重新加入游戏"）。

禁用高优先级中断
-------------------------------------

在正常过程中，偶尔必须使用 ``up_irq_save()`` 内联函数禁用中断，以防止中断级和非中断级逻辑之间共享资源的使用冲突。现在问题来了，如果我们使用 *BASEPRI* 来禁用中断并启用了高优先级中断（``CONFIG_ARCH_HIPRI_INTERRUPT=y``），我们是禁用除 *SVCall* 之外的所有中断（我们无法禁用 *SVCall* 中断）吗？还是只禁用"普通"中断？

如果我们使用 *BASEPRI* 寄存器来禁用中断，那么答案是我们必须只禁用普通中断。这是因为我们无法禁用 *SVCall* 中断，并且我们不能允许 *SVCall* 中断以比高优先级中断更高的优先级运行。否则，它们将在高优先级中断响应时间中引入抖动。

因此，如果你需要禁用高优先级中断，你将不得不在产生中断的外设或中断控制器 *NVIC* 处禁用中断。通过 *BASEPRI* 寄存器禁用全局中断不得影响高优先级中断。

依赖项
------------

* ``CONFIG_ARCH_HAVE_IRQPRIO``。必须启用对优先级中断支持的支持。

* 浮点寄存器。如果与支持硬件浮点的 Cortex-M4 一起使用，你不能在高优先级中断处理程序中使用硬件浮点，除非你使用支持在所有中断上保存浮点寄存器的通用向量逻辑。

配置高优先级中断
------------------------------------

如何指定高优先级中断？你需要做两件事：

首先，你需要更改向量表中的地址，使高优先级中断分派到你的特殊 C 中断处理程序。有两种方法可以做到这一点：

* 如果你选择 ``CONFIG_ARCH_RAMVECTORS``，向量将保存在 RAM 中，系统将支持接口：``int up_ramvec_attach(int irq, up_vector_t vector)``。该接口可用于在运行时将你的 C 中断处理程序附加到向量。

* 或者，你可以将向量保存在 FLASH 中，但为此，你必须开发自己的自定义向量表。

其次，你需要使用标准接口将中断的优先级设置为 *NVIC* 中的 ``NVIC_SYSH_HIGH_PRIORITY``：``int up_prioritize_irq(int irq, int priority);``
