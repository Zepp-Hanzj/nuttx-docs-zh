类 Unix 文件系统中的特殊文件
=======================================

在类 Unix 操作系统中，`特殊文件` 是设备驱动程序的接口，它在文件系统中显示为普通文件。特殊文件是内置于类 Unix 文件系统（如 EXT2）中的功能。

另见 `维基百科文章 <https://en.wikipedia.org/wiki/Device_file>`_。


其他文件系统中的特殊文件
===================================

一些其他文件系统，如 NTFS，也有一些更有限和不兼容的特殊文件概念。其他文件系统，如 FAT，没有任何这样的概念。在这些其他文件系统上运行的类 Unix 环境（如 Cygwin）必须以不同的方式做事：它们必须使用所操作文件系统的资源来模拟特殊文件。这可能意味着创建具有特殊命名约定或特殊内容的 `常规` 文件，可用于模拟特殊文件的行为。

特殊文件和 NuttX
=======================

NuttX 以非常不同的方式处理事情。任何文件系统都不支持特殊文件。相反，特殊文件只能存在于 NuttX :doc:`伪文件系统 <pseudofs>` 中。这是在初始设计中做出的决定，旨在为资源有限的平台简化事情，同时仍然提供大部分标准的类 Unix/POSIX 编程环境。

NuttX 伪文件系统中特殊文件的优势是什么？减少资源使用，降低启动需求。缺点是什么？在 NuttX 中，特殊文件只能驻留在伪文件系统中。

我意识到的唯一其他后果是 NuttX 无法支持 POSIX 对 ``struct stat`` 结构中 ``st_dev`` 字段的要求。


设备文件和设备编号
===============================

在类 Unix 系统中，设备通过特殊设备文件访问。在类 Unix 系统中，设备文件可以驻留在任何兼容的文件系统中，但按照惯例，总是放在 ``/dev`` 目录中。NuttX 通过顶层 `root` 文件系统是伪文件系统且 ``/dev`` 是伪文件系统的一部分来实现与此约定的编程兼容性。

但还有一个更大的区别。更大的区别在于设备驱动程序如何注册和如何访问。类 Unix 设备文件的主要内容只是一个数字，一个设备号，通常表示为 ``dev_t`` 类型。设备号是一个编码，由 `主` 设备号和 `次` 设备号组成。主设备号标识驱动程序的类型，次设备号标识该类型驱动程序的一个实例。

从文件系统的角度来看，这些数字没有什么特别的。即使设备文件存在具有特定主次号的设备文件，也不意味着实际上有任何真正的驱动程序实例支持该设备文件。例如，您可以从 Linux 命令行创建设备文件，只需知道主次设备号：

.. code-block:: C

    dev_t makedev(unsigned int maj, unsigned int min);

在类 Unix 系统中，当您尝试打开设备驱动程序时，必须发生几件事：系统必须打开设备文件，获取设备主次号，然后在某些已注册设备驱动程序的内部 `注册表` 中查找驱动程序实例。如果找到一个，则系统可以完成打开操作。

因此，设备号是设备驱动程序 `注册表` 的某种 `键`。NuttX 的做法非常不同：没有设备号，伪文件系统 `就是` 设备注册表！在 NuttX 中，用户不能创建设备文件；它们只能由设备驱动程序通过调用以下内部接口来创建：

.. code-block:: c

    int register_driver(FAR const char *path, FAR const struct file_operations *fops, mode_t mode, FAR void *priv);

``path`` 参数决定设备文件应放置在伪文件系统中的位置。``mode`` 提供设备文件权限。``fops`` 和 ``priv`` 提供注册表的内部信息。

因此，在 NuttX 中打开设备驱动程序时，涉及的步骤要少得多：系统仍然必须打开设备文件，但由于伪文件系统 `就是` 设备注册表，所有设备驱动程序信息都可用，``open`` 操作无需进一步操作即可完成。

命名资源
===============

在 NuttX 中使用伪文件系统来管理设备文件与 NuttX 设备核心哲学一致：NuttX VFS 和伪文件系统特别用于管理所有命名的 OS 资源。这不仅适用于设备文件和其他特殊文件，还适用于命名消息队列和命名信号量（可以在伪文件系统的 ``/var`` 目录中找到）。
