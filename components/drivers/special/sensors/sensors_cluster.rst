.. warning:: 此接口在 NuttX 中不常用，不建议用于实现新传感器。
             将来应将其删除。

通用传感器寄存器接口
================================

由 Bob Feretich 贡献

背景和问题陈述：

现代传感器的功能和性能已经大幅增长。大多数传感器现在都能够实现
某种程度的自主行为，有些还允许用户在其中加载固件并作为纳米控制器
运行。其他传感器具有非常复杂的内置数字滤波器，可以用数百个参数
进行编程。

目前，NuttX drivers/sensors 目录中的大多数传感器驱动程序实现了
file_ops open()、close() 和 read() 函数。open() 函数初始化传感器
并将其置于可以以默认配置传输实时数据的模式。close() 函数将传感器
置于低功耗关闭模式。read() 函数从传感器最常用的数据输出寄存器
返回最新的数据样本。write() 函数很少实现，即使实现也没有一致性。
lseek() 和 poll() 函数似乎完全被忽略。这导致传感器仅以最原始的
模式运行，使用固定的"默认配置"。

为了解决这个问题，传感器驱动程序实现了 ioctl() 函数来执行配置、
编程传感器和管理自主活动。Ioctl 提供了一种方法，使用户程序可以
通过高级驱动程序隧道来访问和控制设备特定的功能。使用 ioctl 的
问题是，在使用 ioctl 接口之前，必须先打开传感器驱动程序；而 open()
函数会导致驱动程序开始执行这些原始操作，因此在 ioctl 可以按需管理
驱动程序之前，必须先使用 ioctl 来撤消 open() 函数引起的通用操作。
另一个主要问题是传感器之间的 ioctl 定义没有一致性，即使是最常见的
传感器操作（如写入传感器控制寄存器或读取传感器状态寄存器）也没有。

目的：

"通用传感器寄存器接口"的目的是实现 file_ops 接口的一致和更有用的
定义，并使 file_ops open() 函数在建立传感器的初始操作状态时更加
灵活。将保持与实现当前 open()、close()、read() 接口的用户应用
程序的兼容性；并且现代传感器的更强大功能将通过此接口变得可访问。

范围：

适用于 I2C 和 SPI 连接的传感器，以及一些串口连接的传感器。

file_ops 接口定义：

open()：此函数执行以下操作...

  1) 读取传感器 ID 寄存器。如果传感器响应意外值，则...

     a) 驱动程序的 write() 函数被禁用。
     b) open 函数初始化驱动程序实例，以便可以执行 read() 和
        lseek() 操作以进行问题诊断，但不初始化传感器硬件。
        （不向传感器执行写入操作。）
     c) errno 全局变量设置为正的 ENODEV（"无此设备"）。
     d) open() 函数成功返回 file_handle。
         请注意，调用例程应在调用 open() 之前清除 errno。
         （file_ops 规则禁止驱动程序将 errno 设置为零。）

  2) 其他 file_ops 函数被启用。
  3) 驱动程序的"当前寄存器地址"状态变量设置为传感器的第一个
     传感器数据输出寄存器。（这将使 read() 调用返回实时传感器
     数据，并保持与现有用户程序的兼容性。）
  4) 如果驱动程序支持默认工作任务，并且在传感器配置结构中指定了
     中断处理程序，则默认工作任务被绑定到默认工作任务。
  5) 检查传感器配置结构（提供给驱动程序注册函数的）以确定是否
     指定了自定义传感器配置。（自定义配置基本上是一个
     (device_reg_address, value) 对数组，通过"单寄存器写入"操作
     写入传感器。如果指定了自定义传感器配置，则将该配置写入
     传感器，否则将"默认传感器配置"写入传感器。
     （写入此数据的副作用可能导致中断发生和数据被传输到/
     从工作任务。）
  6) open() 函数成功返回 file_handle。

``close()``：此函数停止传感器活动并将其置于低功耗模式。此传感器
  驱动程序实例的 file_ops 接口函数被禁用。（open() 除外）

``read()``：此函数的操作取决于是否正在运行"默认工作任务"以及
  驱动程序的"当前寄存器地址"状态变量的值。

  如果正在运行"默认工作任务"，

    并且驱动程序的"当前寄存器地址"等于第一个传感器数据输出
      寄存器的值，
    并且要读取的字节数小于或等于"默认工作任务"样本中的字节数，

  则数据从"默认工作任务"的样本内存复制到调用者提供的缓冲区。

  否则，此函数将数据从传感器寄存器传输到调用者提供的数据缓冲区。
  读取的第一个字节来自传感器"当前寄存器地址"指定的传感器寄存器
  地址。后续要读取的字节的地址是上下文敏感的。如果需要多次总线
  传输来完成读取，则将使用"多字节"（有时称为"突发模式"）数据
  传输来填充缓冲区。
  请参阅传感器的数据手册以确定"多字节"数据传输的自动递增行为。

  请注意：大多数传感器每个样本只收集几个字节的数据。在高速总线
  （如 SPI 和一些高速 i2c 和串行接口）上发生的小数据传输直接从
  传感器硬件收集比使用工作任务作为中介更有效。

``write()``：此函数将数据从调用者提供的数据缓冲区传输到传感器
  寄存器。写入的第一个字节是到传感器"当前寄存器地址"指定的传感器
  寄存器地址。后续要读取的字节的地址是上下文敏感的。如果需要多次
  总线传输来完成写入，则将使用"多字节"（有时称为"突发模式"）
  数据传输来从缓冲区传输数据。

  请参阅传感器的数据手册以确定"多字节"数据传输的自动递增行为。

  请注意：如果 write() 函数被禁用，则不会执行写入，函数将返回
  0（传输的字符数），errno 设置为 -EROFS（"只读文件系统"）。

``lseek()``：此函数设置传感器的"当前寄存器地址"（seek_address）的
  值。open() 函数将"当前寄存器地址"初始化为第一个传感器数据输出
  寄存器，因此除非用户需要更改传感器配置，否则不需要调用 lseek()。
  read() 和 write() 都不会更改传感器的"当前寄存器地址"。

  lseek 的定义为...::

      off_t lseek(int fd, off_t offset, int whence);

  对于 whence == SEEK_SET，传感器的"当前寄存器地址"将设置为 offset。

  对于 whence == SEEK_CUR，offset 将被添加到传感器的"当前寄存器
  地址"。

  对于 whence == SEEK_END，offset 被忽略，传感器的"当前寄存器
  地址"设置为第一个传感器数据输出寄存器。

  如果结果"当前寄存器地址"对传感器无效，lseek() 将返回错误。

``ioctl()``：Ioctl() 仍然可以使用，此接口不尝试对其进行规范。
  但是，预计需要的 ioctl 会少得多。

上述接口可用于根据应用需求完全配置传感器，包括将固件加载到
传感器状态机的能力

传感器集群驱动程序接口
===============================

由 Bob Feretich 贡献

背景和问题陈述：

大多数微控制器可以支持 8 MHz 或更高的 SPI 总线传输。大多数 SPI
连接的传感器可以支持 10 MHz SPI 总线。大多数三轴加速度计、三轴
陀螺仪或三轴磁力计每个样本只使用 6 个字节。许多传感器每个样本
使用少于 6 个字节。在 8 MHz SPI 总线上传输 6 字节样本大约需要
8 微秒。（此时间包括一个命令字节、6 个数据字节以及片选设置和
保持。）因此，对于以下讨论，请记住我们想要执行的传感器样本收集
工作理想情况下每个样本应花费 8 微秒。

drivers/sensors 目录中的驱动程序仅支持用户空间 file_ops 接口
（通过 POSIX open/read/close 函数使用文件描述符访问驱动程序）。
此外，这些驱动程序通常启动自己的工作任务来执行传感器数据收集，
即使它们的传感器每个样本只传输几个字节的数据，并且这些传输是
通过高性能总线进行的。

使用当前实现...

1) 传感器"数据就绪"或定时器中断发生。
2) 上下文被保存，驱动程序的中断处理程序被计划运行。
3) NuttX 调度程序调度驱动程序的中断处理程序任务。
4) 驱动程序的中断处理程序任务向驱动程序工作任务正在等待的信号量
   发送信号。
5) NuttX 恢复驱动程序工作任务的上下文并开始运行。
6) 驱动程序工作任务启动 I/O 以收集样本。（这是执行 8 微秒实际
   工作的地方。）并等待 SPI 数据传输完成信号量。
7) NuttX 保存驱动程序工作任务的上下文，调度程序在等待期间调度
   其他任务运行。请注意，这是一件好事。此任务可能正在执行其他
   实际工作。我们希望在数据传输期间发生这种情况。
8) 数据传输完成导致中断。NuttX 保存当前上下文并恢复驱动程序
   工作任务的上下文。
9) 驱动程序工作任务进入休眠，等待下一个传感器"数据就绪"或
   定时器中断的信号量。
10) NuttX 保存驱动程序工作任务的上下文，调度程序在等待期间
    调度其他任务运行。

独立于上述...

a) 传感器应用程序程序执行 file_ops read() 以收集样本。
b) NuttX 高级驱动程序接收控制，执行薄层维护并调用传感器驱动
   程序的 read 函数。
c) 传感器驱动程序的 read 函数将最新样本从工作任务的数据区域
   复制到应用程序的缓冲区并返回。
d) NuttX 高级驱动程序接收控制，执行薄层维护并返回。
e) 应用程序处理样本。

使用 216 MHz STM32F7 且没有其他活动发生时，我们计时上述过程的
平均耗时为 45 微秒。

大多数传感器应用程序处理来自多个传感器的数据。（9-DoF IMU 通常
表示为三个传感器（加速度计、陀螺仪和磁力计）。在这种情况下，
1-10 的三个副本并行发生。

在使用实时数据的应用中，此方法的上下文切换抖动和缓存污染会严重
影响系统性能。在使用传感器 FIFO 数据的应用中，因此每次迭代收集
大量数据，数据收集的非"零拷贝"性质成为性能问题。

目的：

"传感器集群驱动程序接口"为应用程序提供了一种标准机制，以更高效的
方式从多个传感器驱动程序收集数据。它显著减少了运行任务的数量以及
它们引起的上下文抖动和缓存污染。它还允许传感器数据的"零拷贝"收集。

传感器集群驱动程序接口使用单个"工作任务"由任意数量的驱动程序
共享。此共享工作任务是一个内核任务，像驱动程序一样注册，支持
到应用程序程序的驱动程序接口，并从多个传感器（传感器集群）收集
数据，我们称之为"传感器集群驱动程序"。

其目标是将上面详述的事件序列更改为...

1) 传感器"数据就绪"或定时器中断发生。
2) 上下文被保存，集群驱动程序的中断处理程序被计划运行。
3) NuttX 调度程序调度集群驱动程序的中断处理程序任务。
4) 集群驱动程序的中断处理程序任务向集群驱动程序工作任务正在
   等待的信号量发送信号。
5) NuttX 恢复驱动程序工作任务的上下文并开始运行。
6) 集群驱动程序工作任务启动 I/O 以收集样本。这里有两个选择。
   编程 I/O（PIO）或 DMA。如果 PIO 对小样本大小最快，但它将在
   传输的整个持续时间内锁定处理器；它一次只能从一个传感器传输；
   工作任务应偶尔手动让出控制权以允许其他任务运行。DMA 具有更高
   的启动和完成开销，但对于长传输更快，可以同时从不同总线上的
   传感器执行传输，并在传输发生时自动释放处理器。因此，我们的
   驱动程序允许工作任务在 PIO（driver_read()）和 DMA
   （driver_exchange()）之间选择，这是
   sensor_cluster_operations_s 结构的常见扩展。因此，无论哪种
   方式，在一次或多次传输后，我们让出控制权并转到下一步。请注意，
   数据正在直接传输到应用程序程序提供的缓冲区中；因此不需要执行
   复制。
7) NuttX 保存集群驱动程序工作任务的上下文，调度程序在等待期间
   调度其他任务运行。再次请注意，这是一件好事。此任务可能正在
   执行其他实际工作。我们希望在数据传输期间发生这种情况。
8) 上一次数据传输的最后一次完成导致中断。NuttX 保存当前上下文
   并恢复集群驱动程序工作任务的上下文。如果有更多传感器数据要
   收集，则转到步骤 6。否则，它向将唤醒应用程序的信号量发送信号。
9) 驱动程序工作任务进入休眠，等待下一个传感器"数据就绪"或
   定时器中断的信号量。
10) NuttX 保存驱动程序工作任务的上下文，调度程序在等待期间
    调度其他任务运行。

独立于上述...

a) 传感器应用程序程序执行 file_ops read() 以收集样本。
b) NuttX 高级驱动程序接收控制，执行薄层维护并调用传感器驱动
   程序的 read 函数。
c) 传感器驱动程序的 read 函数将最新样本从工作任务的数据区域
   复制到应用程序的缓冲区并返回。
d) NuttX 高级驱动程序接收控制，执行薄层维护并返回。
e) 应用程序处理样本。

因此，从三个传感器收集数据时，此机制节省了...

* 处理 2 个传感器"数据就绪"或定时器中断（步骤 1 - 4）。
* 2 次唤醒和调度工作任务（步骤 5）。
* 2 次到其他任务的上下文切换（步骤 9 和 10）
* 如果三个传感器在不同的总线上，则 2 次

步骤 6 - 8 也可能被节省。

* 收集的传感器数据的额外复制操作。
* 2 个竞争工作任务引起的缓存污染。

定义：

"叶子驱动程序" - 实现"传感器集群驱动程序接口"的内核驱动程序，
  以便可以被集群驱动程序调用。

"集群驱动程序" - 使用"传感器集群驱动程序接口"调用叶子驱动程序
  的内核驱动程序。

"入口点向量" - 函数地址数组，叶子驱动程序将允许集群驱动程序调用。

"叶子驱动程序实例句柄" - 指向不透明叶子驱动程序结构的指针，
  用于标识叶子驱动程序的实例。叶子驱动程序在注册期间将其存储
  在配置结构中。

传感器集群接口描述：

* 入口点向量的定义。这类似于提供给 file-ops 高级驱动程序的入口点
  向量。此入口点向量必须将 sensor_cluster_operations_s 结构作为
  其第一个成员。
* 叶子驱动程序配置结构中的驱动程序入口点向量成员的定义。叶子
  驱动程序注册函数必须将其入口点向量的地址存储在此字段中。
* 叶子驱动程序配置结构中的实例句柄成员的定义。叶子驱动程序
  注册函数必须将正在注册的叶子驱动程序实例的句柄（不透明指针）
  存储在此字段中。请注意，这应与叶子驱动程序提供给 NuttX 以
  注册自身的句柄相同。集群驱动程序将在对叶子驱动程序的调用中
  将此句柄作为参数包含。

.. code-block:: C

   struct sensor_cluster_operations_s
   {
     CODE int     (*driver_open)(FAR void *instance_handle, int32_t arg);
     CODE int     (*driver_close)(FAR void *instance_handle, int32_t arg);
     CODE ssize_t (*driver_read)(FAR void *instance_handle, FAR char *buffer,
                                 size_t buflen);
     CODE ssize_t (*driver_write)(FAR void *instance_handle,
                                  FAR const char *buffer, size_t buflen);
     CODE off_t   (*driver_seek)(FAR void *instance_handle, off_t offset,
                                 int whence);
     CODE int     (*driver_ioctl)(FAR void *instance_handle, int cmd,
                                  unsigned long arg);
     CODE int     (*driver_suspend)(FAR void *instance_handle, int32_t arg);
     CODE int     (*driver_resume)(FAR void *instance_handle, int32_t arg);
   };

请注意，sensor_cluster_operations_s 与 NuttX fs.h file_operations
结构非常相似。这允许当前 file_operations 函数成为这些函数的薄
包装器。

``driver_open()`` 与 fs.h open() 相同，但可以指定 arg 以在传感器
配置和初始操作中提供更大的灵活性。当 arg = 0 时，driver_open()
的功能必须与 open() 相同。

``driver_close()`` 与 fs.h close() 相同，但可以指定 arg 以在选择
传感器低功耗状态时提供更大的灵活性。当 arg = 0 时，driver_close()
的功能必须与 close() 相同。

``driver_read()`` 与 fs.h read() 相同。

``driver_write()`` 与 fs.h write() 相同。可选。如果不支持则设置
为 NULL。

``driver_seek()`` 与 fs.h seek() 相同。可选。如果不支持则设置
为 NULL。

``driver_ioctl()`` 与 fs.h ioctl() 相同。可选。如果不支持则设置
为 NULL。

``driver_suspend()`` 和 ``driver_resume()`` 可选。如果不支持则
设置为 NULL。传感器应用程序通常会节省功耗并将微控制器置于低功耗
休眠状态。似乎适合保留这些位置供将来使用。这些驱动程序入口点存在
于 Linux 和 Windows 中。由于微控制器和传感器每年都在变得更强大，
应该很快就会需要这些入口点。关于如何标准化其使用和实现的讨论
应独立于此驱动文档进行。

请注意，鼓励所有驱动程序在此公共段之外扩展其入口点向量。例如，
工作任务在编程 I/O 和 DMA 数据传输例程之间进行选择可能是有益的。
应鼓励对入口点向量进行不受监管的扩展，以最大化传感器功能的好处。

操作：

板逻辑（configs 目录）将注册集群驱动程序。集群驱动程序将注册它
将调用的叶子驱动程序。这意味着集群驱动程序可以访问叶子驱动程序
的配置结构，并可以通过入口点向量将叶子驱动程序实例句柄作为参数
传递给叶子驱动程序。

板逻辑或应用程序程序可以 open() 集群驱动程序。集群驱动程序 open()
调用叶子驱动程序的 open() 函数。集群驱动程序 open() 或 read()
函数可以启动收集数据的共享工作任务。

集群驱动程序 close() 函数调用叶子驱动程序的 close 函数。
