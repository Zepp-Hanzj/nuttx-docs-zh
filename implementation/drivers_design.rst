=================
OS 驱动设计
=================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

操作系统识别三种类型的驱动程序，这些驱动程序对应用程序可见。其中两种是 POSIX 标准设备驱动程序类型，一种是非标准类型。还有内部操作系统组件也可能被视为驱动程序，或者更准确地说是下半部驱动程序。有关这些的详细信息如下。

字符驱动和块驱动
===========================

标准驱动程序类型包括：

* **字符驱动程序**。首先是字符驱动程序。这些是支持用户通过 ``read()``、``write()`` 等进行访问的驱动程序。其他的则不是自然支持的。字符驱动程序实现传入或传出字节流。

* **块驱动程序**。这些用于支持面向块的 I/O 的文件系统，而非字符流。用户不能*直接*访问块驱动程序。

但是，用户可以通过字符驱动程序代理间接访问块驱动程序。字符驱动程序和块驱动程序都由设备节点表示，通常在 ``/dev`` 中。但如果您尝试打开块驱动程序，会发生一些非常奇怪的事情：会自动实例化一个临时的、无名称的代理字符驱动程序，将字符驱动程序的字节流映射为块，并调解对块驱动程序的驱动程序访问。这是 ``drivers/bch`` 中的逻辑。BCH 代表块到字符。因此，从应用程序的角度来看，两者似乎都是字符驱动程序，应用程序可以以相同的方式与两者交互。

此功能被利用，例如，被 NuttX 文件系统格式化应用程序（如 mkfatfs）用于在块驱动程序上格式化 FAT 系统。

还有补充功能，即循环设备，它将字符驱动程序转换为块驱动程序。循环设备通常用于在 RAM 中格式化文件系统映像。

MTD 驱动
===========

非标准驱动程序是：

* **内存技术驱动程序（MTD）**。此命名借鉴自 ``infradead.org``，但并非源自他们的任何 MTD 逻辑。MTD 驱动程序管理基于内存的设备，如 FLASH 或 EEPROM。MTD FLASH 内存驱动程序与块驱动程序非常相似，但 FLASH 具有一些不同的属性，最显著的是在写入之前必须擦除 FLASH。

MTD 具有与块驱动程序相同的便利性：它们可以作为 ``/dev`` 下的设备节点出现，如果作为字符驱动程序打开，可以被代理为像字符驱动程序一样工作。此外，它们还有一些额外的功能：MTD 驱动程序可以堆叠在一起以扩展低级 MTD 驱动程序的功能。例如，``drivers/mtd/sector512.c`` 是一个 MTD 驱动程序，当分层在另一个 MTD 驱动程序之上时，它会将 FLASH 的页面大小更改为 512 字节。

``drivers/mtd/mtd_partitions.c`` 可用于将大型 FLASH 分割成独立的分区，每个分区看起来像另一个 MTD 驱动程序。

``drivers/mtd/ftl.c`` 也很有趣。FTL 代表 FLASH 转换层。FTL 驱动程序是一个 MTD 驱动程序，当分层在另一个 MTD 驱动程序之上时，将 MTD 驱动程序转换为块驱动程序。排列组合是无穷无尽的。

单体驱动
==================

当人们想到操作系统中的设备驱动程序时，会想到一个单一的东西，一个带有两个主要接口的框图中的单个块：

* 设备单体驱动程序暴露一个标准的设备驱动程序接口。通过**虚拟文件系统（VFS）**，这为应用程序提供了对驱动程序功能的用户界面。以及

* 由设备驱动程序管理的硬件的低级接口。

上半部和下半部驱动
=================================

NuttX 支持许多不同的 MCU 平台，每个平台都有许多相似但不同的内置外设。
当然，我们可以想象一种实现，其中每个这样的外设都由前面段落中描述的单体驱动程序支持。
然而，这将涉及大量代码重复。
MCU 外设在低级寄存器接口级别可能是唯一的。
然而，在更高级别的抽象中，这些外设实际上非常相似。

NuttX 使用*上半部*和*下半部*驱动程序的概念来减少代码和驱动程序开发中的重复。
这样的实现产生两个结果；系统框图中的两个块：一组公共共享驱动程序中的上半部驱动程序，以及 MCU 特定的下半部驱动程序。

如前所述，这两个驱动程序组件中的每一个都有两个功能接口。
对于上半部驱动程序：

* 上半部设备驱动程序暴露一个标准的驱动程序接口。
  通过**虚拟文件系统（VFS）**，这再次为应用程序提供了对驱动程序功能的用户界面。以及

* 下半部接口的上半部侧连接到由下半部设备驱动程序管理的 MCU 特定硬件。

对于下半部驱动程序：

* 连接到上半部驱动程序的下半部接口侧，以及

* 由下半部设备驱动程序管理的硬件的低级接口。

一对多：封装和多态性
---------------------------------------------------

这些模块化的上半部和下半部驱动程序具有某些与面向对象设计相关的属性：封装、数据抽象，当然还有多态性。
由于这种封装，上半部驱动程序完全不知道下半部驱动程序中的任何实现细节。
上半部和下半部驱动程序集成所需的一切都由两者之间定义的接口提供。
事实上，一个上半部驱动程序可以在一对多关系中为多个下半部驱动程序实例提供服务。

例如，某些 MCU 支持 UART、作为 UART 功能的 USART、低功耗 UART（LPUART）和其他可能作为 UART 功能的灵活设备。
每种设备都由一个单独的下半部驱动程序管理，可以在 ``arch/`` 下的相应 ``src/`` 目录中找到。
此外，板上可能有片外的外部 16550 UART 硬件（它有一个通用的下半部驱动程序）。
然而，它们都由位于 ``drivers/serial/serial.c`` 的单一公共串行上半部驱动程序支持。
这只有在下半部驱动程序实现具有类似对象的属性的情况下才有可能。
