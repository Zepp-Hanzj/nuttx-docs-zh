==============================
下半部中断处理程序
==============================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

RTOS 中断
===============

设计良好的 RTOS 依赖于最精简的中断级处理。这与裸机编程的概念非常不同：

* 在裸机编程中，大部分实时工作通常在中断处理程序中执行。中断处理程序的执行时间可能因此中断级处理而显著延长。

为了补偿这种延长的中断处理时间，裸机程序员还需要优先级中断：

* 如果在低优先级中断的扩展处理期间发生更高优先级中断的中断请求，则该中断处理程序本身将被中断以服务更高优先级的中断请求。这样，裸机中断处理是嵌套的。

使用 RTOS 时，实时策略非常不同：

* 中断必须非常、非常简短地运行，以免干扰 RTOS 实时调度。通常，中断只执行任何必要的小型内务处理，然后通过某种进程间通信（IPC）唤醒某个任务来立即延迟处理。RTOS 负责实时行为，而不是中断。以及
* 由于中断必须非常简短，嵌套中断几乎没有或没有收益。

扩展中断处理
==============================

但如果需要扩展的中断处理怎么办？如果有大量与硬件相关的操作必须在将处理交给通用实时任务之前尽快完成怎么办？

在 NuttX 中，这通过一个名为"高优先级工作队列"的高优先级跳板来处理。它是一个跳板，因为它在通知正常实时任务之前更改中断处理上下文以进行扩展中断处理。

在该超高优先级工作线程上处理然后在启用中断的情况下完成扩展中断处理，但不受任何其他实时任务的干扰。

扩展处理完成后，高优先级工作线程可以通过某种 IPC 继续处理到正常的实时任务。

在中断处理程序中执行的禁用中断的中断处理部分称为上半部中断处理；在高优先级工作队列上执行的启用中断的中断处理部分称为下半部中断处理。

高优先级工作队列
========================

NuttX 支持高优先级工作队列以及具有某些不同属性的低优先级工作队列。高优先级工作队列专门用于支持下半部中断处理。高优先级工作队列的其他使用可能不合适，并可能损害系统的实时性能。

高优先级工作队列必须具有以下属性：

* **最高优先级** 高优先级工作队列必须是系统中最高优先级的任务。没有其他任务应以更高优先级执行；不允许任何其他任务干扰高优先级工作队列的执行。

* **零延迟上下文切换** 假设高优先级工作队列的优先级是系统中最高的，那么从上半部中断处理到下半部中断处理不会有上下文切换开销，除了从中断返回的正常开销。从中断返回时，系统将立即向量到高优先级工作线程。

* **简短处理** 高优先级工作队列上的处理仍然必须简短。当高优先级工作器被信号通知时，如果有高优先级工作正在进行，则该处理将被排队并延迟直到可以处理。该延迟将为您的实时响应添加抖动。您绝不能为高优先级工作线程产生积压的工作！

* **无等待** 在高优先级工作队列上执行的工作绝不能等待高优先级工作线程上的资源或事件。在高优先级工作队列上等待会阻塞队列并再次损害实时性能。

设置下半部中断处理
===========================================

下半部中断处理由上半部中断处理调度，只需调用函数 ``work_queue()``：

.. code-block:: C

   int work_queue(int qid, FAR struct work_s *work, worker_t worker,
                  FAR void *arg, clock_t delay);

此相同的接口对于高优先级和低优先级是相同的。qid 参数区分将使用哪个工作队列。对于下半部中断处理，``qid`` 必须设置为 ``HPWORK``。

work 参数是将用于实际排队工作的内存。它对调用者没有意义；它只是调用者的内存分配。否则，工作结构完全由工作队列逻辑管理。调用者不应直接修改工作队列结构的内容。如果在先前的工作被执行并从队列中移除之前调用 ``work_queue()``，则任何待处理的工作将被取消并丢失。可以调用 ``work_available()`` 函数来确定由工作结构表示的工作是否仍在使用中。

对于当前的中断处理情况，工作结构必须是预分配的或静态分配的，因为不支持从中断处理上下文进行动态分配。

``worker`` 是将执行下半部中断工作的函数的名称。
``arg`` 是用户提供的任意值，将在工作函数执行时提供给它。
通常 ``arg`` 产生执行工作的某种上下文。
工作函数的类型由以下给出：

.. code-block:: C

   typedef CODE void (*worker_t)(FAR void *arg);

其中 ``arg`` 与传递给 ``work_queue()`` 的值相同。

处理或工作可以延迟执行。
``work_queue()`` 的 ``delay`` 参数以系统时钟滴答为单位提供该时间延迟。但是，当用于提供下半部中断处理时，延迟应始终为零。
