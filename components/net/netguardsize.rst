====================
CONFIG_NET_GUARDSIZE
====================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

所有驱动的全局选项
=============================

``CONFIG_NET_GUARDSIZE`` 是全局选项。它被添加到每个驱动数据包缓冲区的分配大小中。
目前它是一个非常小的值，默认仅为两个字节。因此它不会占用大量内存，
应将其添加到所有驱动的数据包大小中以实现通用性。但为什么呢？

它应该（最终）变得更大且对所有驱动通用。我们需要了解它现在的用途以及
未来可能的用途。可能涉及的内容比您最初考虑的要多得多。

数据包接收
==============

对于数据包接收，某些硬件需要它，但其他硬件不需要。通常硬件会在数据包末尾
DMA 传输 2 字节的 FCS 或其他硬件特定信息。但这只是整个故事的一部分。
``CONFIG_NET_GUARDSIZE`` 不仅用于硬件数据包接收。

数据包传输
===================

数据包传输存在几个问题。这些问题定义不太明确，需要进一步研究，
但在我们了解如何处理这些问题之前，需要保留所有驱动数据包定义：

* 内存溢出错误

  过去曾存在一个错误，在 TX 消息格式化期间导致写入超过缓冲区末尾几个字节。
  我不知道该错误是否仍然存在，但最小的两字节 ``CONFIG_NET_GUARDSIZE``
  足以消除该错误。这就是它被称为 GUARD（防护）的原因：
  其主要目的是防止溢出数据包缓冲区并破坏后续内存。

  我不知道我们现在是否还有这样的错误。也许仍然存在？也许不存在？
  无论如何，有这样的防护对于可靠性来说是一件好事。

* 可变大小的 IP/TCP 头部

  IP 数据包的格式化方式存在限制。基本上它们是这样格式化的：

    #. 当接收到数据包时，设置指向有效负载位置的指针
       (d_appdata)。这是数据包缓冲区中的偏移量。
       对于 TCP，这考虑了 MAC/Ethernet 头部、最小 IPv4/IPv6 头部大小
       和最小 TCP 头部大小。

    #. TCP 有效负载被写入该位置，
    #. 正确大小的 IPv4/IPv6 头部和正确大小的 TCP 头部被添加在有效负载下方，
       最后
    #. 添加 MAC/Ethernet 头部。

  数据包在数据包中的起始偏移量不再是零，而是数据包缓冲区中的某个可变偏移量。
  必须将该新的起始偏移量传递给驱动才能发送数据包。

  使这一切工作的关键是：

    * 在所有驱动缓冲区中保留 ``CONFIG_NET_GUARDSIZE``，以及
    * 将 ``CONFIG_NET_GUARDSIZE`` 设置为 IPv4/IPv6 和 TCP 选项的最大大小
      （取决于启用了哪个 IP 版本以及是否启用了 TCP）
    * 扩展驱动接口以接受驱动数据包缓冲区中的数据偏移量。

* 可变 MSS

  与此密切相关的是 MSS，即有效负载的最大大小。目前这是一个常数，
  因为它假设最小头部长度。它应该是可变的，取决于实际的头部大小。
