====================
开发工作流程
====================

NuttX 的开发工作流程基于以 GitHub Pull Request (PR) 形式提交的贡献。
无论是外部贡献者还是 NuttX 维护者都是如此，因为一般规则不允许直接推送到
仓库。提交后，您的 PR 将通过持续集成 (CI) 实践进行审查和检查。

您应该了解以下内容：

  - 所有贡献必须遵守 :doc:`编码标准 <coding_style>`。
    您可以使用 ``nxstyle`` 检查您的文件，或使用 ``checkpatch`` 脚本
    检查完整的补丁集（两者都位于 NuttX 仓库的 ``tools`` 子目录中）。
    此检查也将在 CI 期间自动运行以确保一致性。

    请注意，并非仓库中的所有现有文件都已符合标准，这是一项持续的工作。
    因此，如果您向现有文件提交补丁，您可能需要使该文件符合标准，
    即使您不是这些标准违规的责任人。

    也请您将任何样式修复与功能更改分开到单独的提交中，
    以便在审查期间更容易阅读。

  - 在开始任何非平凡的贡献工作之前，请订阅邮件列表并询问您的想法，
    以避免走错误路线而浪费精力。

  - 如果您提交的是原创贡献（您自己从头编写的代码），
    它必须根据 Apache 2.0 许可证的条款提交，
    使用相应的 :ref:`头文件 <contributing/coding_style:Appendix>`。

    请注意，如果您是公司的雇员，版权通常属于公司，
    因此这意味着公司必须授权并提交适当的许可协议。

  - 如果您提交的是第三方代码：

    - 活跃开发项目的代码不被接受包含在 NuttX 中（即：创建分支）。
      预计第三方代码中为 NuttX 支持所需的更改将在这些项目中实现。
      作为中间解决方案，可以包含一个应用于此第三方代码的补丁，
      该补丁将在构建过程中被拉取。

    - 如果来自不活跃的项目，在许可条款允许且被认为有足够价值的情况下，
      可以考虑将其包含在 NuttX 中，但需要考虑此后必须在 NuttX 中维护这些代码。

      请注意，不希望在仓库中包含非 Apache 2.0 许可的代码，
      即使许可本身允许（例如 BSD 许可证）。
