问题
========

由于某些未知原因，我看到较差的 TCP 网络性能。

答案
======

首先让我们谈谈 TCP 发送性能。

性能瓶颈来源
---------------------------------

通用 TCP 发送性能并非由 TCP 协议栈决定，而是更多地取决于网络设备驱动。
不良的网络性能是由于数据包传输*之间*损失的时间。数据包传输本身以
线速*进行。因此，如果您想改善给定网络的性能，必须减少传输之间的时间损失。
没有其他方法。

忽略以太网问题，如冲突、退避延迟、包间间隔 (IPG) 等。

数据包之间的时间主要受限于网络驱动的缓冲设计。如果您想提高性能，
则必须改善网络驱动的缓冲。您需要支持许多全尺寸（1500 字节）数据包缓冲区。
您必须能够查询网络以获取要传输的新数据，并将这些传输排队到数据包缓冲区中。
为了达到峰值性能，网络驱动必须在前一个传输完成之前准备好下一个传输缓冲区，
以最小化数据包传输之间的间隙。

不同的网络设备也支持或多或少的高效接口：性能最差的支持一次只能处理
一个数据包的接口，性能最好的能够保留在内存中的数据包缓冲区链表，
并对一系列数据包执行 scatter-gather DMA。

在 NuttX TCP 协议栈中，您还可以通过启用 TCP 写缓冲来提高性能。
但驱动才是真正关键。

对网络协议栈性能进行深入分析以识别瓶颈并产生性能改进的想法将是很好的。
没有人做过。如果我意识到任何与协议栈相关的性能问题，我一定会解决它。

RFC 1122
--------

NuttX TCP 中缺少一个重要功能，该功能在没有写缓冲时可以提供帮助：
没有写缓冲时，send() 在接收方 ACK 之前不会返回。
但根据 RFC 1122，主机不必立即 ACK 每个数据包；
主机可以在 ACK 前等待 500 毫秒。当向 RFC 1122 客户端进行小型、
非缓冲传输时，这种组合可能导致非常慢的性能。但是，RFC 1122 必须
至少每隔一个（奇数）数据包进行 ACK，因此启用写缓冲的数据包序列
不会受到此问题的影响。

`更新：RFC 1122 支持已于 2019 年 12 月通过提交 66ef6d143a627738ad7f3ce1c065f9b1f3f303b0
添加到 NuttX TCP 协议栈中。不过，这仅影响接收数据包的 ACK 行为，
对发送数据包的性能没有影响；仍然建议使用写缓冲。`

TCPBlaster
----------

我在 ``apps/examples/tcpblaster`` 创建了一个新的测试应用程序来测量 TCP 性能，
并为恰好在我桌面上的配置收集了一些数据。`tcpblaster` 测试为您提供
``Kb/sec`` 的读写传输速率（我不会提到具体数字，因为我认为它们不会
适用于任何其他设置，因此会产生误导）。

有一个不错的 `TCP 吞吐量工具 <https://www.switch.ch/network/tools/tcp_throughput/>`_
提供了一些理论上的性能上限。该工具需要知道 ``MSS``（即您配置的以太网数据包大小
减去以太网头部大小 14）、往返时间 (``RTT``)（毫秒）（您可以从 Linux 主机的 ping
获取）和一个损失常数（我保留了默认值）。使用这些值，我可以确定 NuttX TCP 协议栈
的吞吐量大约在理论极限。您不应该能做得比这更好（实际上，它的性能高于理论极限，
但我认为这就是为什么它是"理论"的）。

因此，如果您对网络性能不满意，我建议您运行 `tcpblaster` 测试，
使用该数据（加上来自 ping 的 ``RTT``）与
`TCP 吞吐量工具 <https://www.switch.ch/network/tools/tcp_throughput/>`_。
如果您仍对性能不满意，请不要立即指责协议栈（每个人都会这样做）。
相反，您应该专注于优化网络配置设置和检查 MCU 中以太网驱动的缓冲处理。

如果您确实发现了协议栈的任何重大性能问题，我当然会很乐意帮助您解决。
或者如果您对改进性能有想法，我也很乐意听取。

接收性能如何？
-------------------------------

以上讨论都涉及*发送性能*，即"我们能以多快的速度通过网络发送数据？"
另一面是接收性能。接收性能是非常不同的事情。在这种情况下，
远程对端完全控制数据包出现在网络上的速率，因此负责所有的原始比特传输速率。

然而，我们也可以将性能重新定义为`成功`传输的字节数。为了使字节成功传输，
它们必须在 NuttX 目标上成功接收和处理。如果数据包`丢失`或`丢弃`，我们就会失败。
如果网络驱动在发送时未准备好接收数据包，则数据包会丢失。
如果数据包被网络丢弃，则是因为它被接收但无法处理，
要么是因为数据包存在某些逻辑问题（这里不是这种情况），
要么是因为我们没有空间来缓冲新接收的数据包。

如果 TCP 数据包丢失或丢弃，则代价很大：该数据包不会被 ACK，
远程对端可能会发送一些乱序数据包，这些数据包也会被丢弃。
最终，远程对端将超时并从丢失的数据包点重新传输数据。

TCP 协议中有帮助管理这些数据溢出的逻辑。TCP 头部包含一个 TCP
`接收窗口`，它告诉远程对端接收方能够缓冲多少数据。
此值在每个接收数据包的 ACK 中发送。如果调整得当，此接收窗口
可能防止因缺少预读存储而丢失数据包。这稍微好一些。
远程对端将暂停发送数据，而不是超时和重新传输。但这仍然是性能损失；
暂停导致的数据包传输间隙将导致传输速率降低。

因此，良好接收的关键是缓冲和处理时间。缓冲同样适用于驱动中的处理，
但与发送性能不同，这通常不是瓶颈。还有一个 NuttX 配置选项控制
TCP 数据包的`预读`缓冲。驱动中的缓冲必须优化以避免丢失数据包；
预读缓冲可以调整以最小化因没有空间缓冲而丢弃的数据包数量。

但接收性能的关键是处理延迟的管理。小的处理延迟可能出现在网络驱动或
TCP 协议栈中。但处理延迟的主要来源是应用程序，它是传入数据的最终消费者。
例如，想象一个 FTP 应用程序正在通过 TCP 接收文件并将文件写入 FLASH 存储器。
这里的主要瓶颈将是写入 FLASH 存储器，这超出了软件的控制范围。

当处理延迟跟上传入数据包的速率时，我们获得最佳接收性能。
如果处理数据速率甚至比接收数据速率慢一点，那么将有不断增长的
缓冲传入数据`积压`需要处理。如果此积压继续增长，最终我们的数据
缓冲能力将耗尽，数据包将被暂停或丢弃，性能将恶化。
在高端远程对端与低端嵌入式系统交互的环境中，由于嵌入式系统有限的
缓冲空间、较低的处理能力和较慢的存储外设，远程对端可以轻松地
溢出嵌入式系统。
