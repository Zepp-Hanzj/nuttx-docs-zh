uIP 和 NuttX
=============

NuttX IP 协议栈的核心源自 Adam Dunkel 的小型 `uIP 协议栈
<http://sourceforge.net/projects/uip-stack/>`_，回溯到版本 1.0。
NuttX TCP/IP 协议栈包含 uIP TCP 状态机和一些 uIP 的"做事方式"，
但除此之外，这两种设计之间现在几乎没有共同之处。

**注意**：uIP 也内置于 Adam Dunkel 的
`Contiki <http://contiki.sourceforge.net/docs/2.6/a01793.html>`_
操作系统中。

uIP、延迟 ACK 和分片数据包
====================================

在 uIP 中，TCP 数据包一次发送一个并进行 ACK。
也就是说，在发送一个 TCP 数据包后，在接收方 ACK 前一个数据包之前
不能发送下一个数据包。
TCP 协议当然支持发送多个数据包，这些数据包可以被接收方异步 ACK。
这种一次一个数据包的逻辑是 uIP 设计中的一种简化；因此，
uIP 只需要一个数据包缓冲区，您可以在最微小的环境中使用 uIP。
这对 uIP 的目标来说是一件好事。

如果您从 Adam Dunkel 的 uIP 升级到他的 `lwIP
<http://savannah.nongnu.org/projects/lwip/>`_ 协议栈，
数据包缓冲的改进是您获得的本质改进。您付出的代价是内存使用。

这种一次一个的数据包传输确实为 uIP 创建了性能问题：
RFC 1122 规定主机可以延迟 ACK 数据包最多 500 毫秒，
但必须对每第二个段响应 ACK。
在基准 uIP 中，这实际上在传输到采用此延迟 ACK 策略的接收方的每个数据包之间
添加了半秒延迟！

uIP 有一个选项可以解决这个问题：
它有一个可以启用的逻辑，将每个数据包分成两半，
在每个数据包中发送一半的数据。
发送更多、更小的数据包听起来不像是性能改进。
这会欺骗遵循 RFC 1122 的接收方接收两个连续的小数据包
并立即 ACK 第二个。
参考：`uip-split.c <http://contiki.sourceforge.net/docs/2.6/a00427_source.html>`_
和 `uip-split.h <http://contiki.sourceforge.net/docs/2.6/a00428.html>`_。

NuttX TCP/IP 协议栈与延迟 ACK
=====================================

NuttX 的底层 TCP/IP 协议栈没有 uIP TCP/IP 协议栈的限制。
它可以发送大量 TCP/IP 数据包，无论它们是否已被 ACK。
这是因为在 NuttX 中，对哪些数据包已被 ACK 和哪些未被 ACK 的记账
已被移动到架构的更高层。

NuttX 在底层 TCP/IP 协议栈之上包含标准的 BSD 套接字接口。
ACK 记账正是在这个更高层的套接字层中完成的，
具体在函数 `send() <http://pubs.opengroup.org/onlinepubs/009695399/functions/send.html>`_
中。如果您通过 ``send()`` 发送大型、多数据包缓冲区，
它将被分解为单独的数据包，每个数据包将尽可能快地发送，
而不关心前一个数据包是否已被 ACK。

但是，NuttX 的 ``send()`` 函数在最终数据包被 ACK 之前不会返回给调用方。
这样做是为了确保调用方的数据已成功发送（或未成功）。
这种行为意味着如果发送了奇数个数据包，在最终数据包之后
``send()`` 收到 ACK 并返回之前仍可能有延迟。

因此 NuttX 的方法类似于 uIP 的做事方式，但确实添加了一个额外的缓冲区，
即用户为 ``send()`` 提供的缓冲区，可用于提高 TCP/IP 性能
（当然，为了符合 ``send()`` 规范，也需要此用户提供的缓冲区）。

NuttX 分片数据包配置
====================================

但如果所有用户缓冲区都小于一个 TCP 数据包的 MSS 会怎样？
假设 MTU 为 1500，而用户 I/O 缓冲区仅为 512 字节？
在这种情况下，``send()`` 性能退化到与 uIP 相同的行为：
在 ``send()`` 可以返回之前以及在可以再次调用 ``send()`` 发送
下一个数据包之前，需要对每个数据包进行 ACK。

修复方法是什么？Yan T 最近贡献了一个修复，
其工作方式类似于 uIP 分片数据包逻辑：
在 ``send()`` 中，逻辑通常在有机会时尝试发送一个完整的数据包。
但是，如果定义了配置选项 ``CONFIG_NET_TCP_SPLIT=y``，
``send()`` 的行为将按以下方式更改：

* ``send()`` 将跟踪`偶数`和`奇数`数据包；`偶数`数据包是我们
  不期望被 ACK 的数据包，`奇数`数据包是我们期望被 ACK 的数据包。
* ``send()`` 然后将根据需要减小偶数数据包的大小，
  以确保始终发送偶数个数据包。每次调用 send 都将导致
  发送偶数个数据包。

这个巧妙的解决方案以与 uIP 分片逻辑相同的方式欺骗 RFC 1122 接收方。
因此，如果您正在与遵循 RFC 1122 ACK 行为的主机合作，
并且您的 MSS 大小大于用户缓冲区的平均大小，
那么启用 ``CONFIG_NET_TCP_SPLIT=y`` 可能会大大提高您的吞吐量。

注意：NuttX `不是` RFC 1122 接收方；NuttX 将 ACK 它接收到的每个 TCP/IP 数据包。

写缓冲
===============

延迟 ACK 问题的最佳技术解决方案是支持`写缓冲`。
写缓冲通过 ``CONFIG_NET_TCP_WRITE_BUFFERS`` 启用。
如果选择此选项，NuttX 网络层将在系统初始化时预分配几个写缓冲区。
然后发送数据缓冲区的工作方式如下：

* ``send()`` (1) 从空闲列表获取预分配的写缓冲区，然后 (2) 将用户
  希望发送的数据缓冲区简单复制到分配的写缓冲区中。
  如果没有可用的写缓冲区，``send()`` 将不得不阻塞等待空闲写缓冲区空间。
* ``send()`` 然后 (3) 将写缓冲区添加到 TCP 套接字的传出数据队列中。
  每个打开的 TCP 套接字都必须支持这样的队列。
  ``send()`` 然后可以 (4) 向调用方返回成功（即使传输仍可能稍后失败）。
* ``send()`` 实现外部的逻辑管理从写缓冲区进行的实际数据传输。
  当以太网驱动能够在 TCP 连接上发送数据包时，
  此外部逻辑 (5) 从写缓冲区复制数据包，以便以太网驱动执行传输
  （`零拷贝`实现更可取）。注意数据必须暂时保留在写缓冲区中；
  它可能需要被重新传输。
* 此外部逻辑还将管理接收 TCP ACK。
  当 TCP 对端确认接收数据时，已确认的数据部分可以 (6) 最终
  从写缓冲区中删除。

以下选项配置 TCP 写缓冲：

* ``CONFIG_NET_TCP_WRITE_BUFSIZE``：一个 TCP 写缓冲区的大小。
* ``CONFIG_NET_NTCP_WRITE_BUFFERS``：TCP 写缓冲区的数量
  （可以为零以禁用 TCP/IP 写缓冲）

NuttX 还支持 TCP 预读缓冲。此选项通过 ``CONFIG_NET_TCP_READAHEAD`` 启用。
TCP 预读缓冲在 TCP 连接上是必需的；否则在没有 ``recv()`` 等待时
接收的数据将丢失。为了一致性，最好使 TCP 写缓冲区实现
以与现有 TCP 预读缓冲类似的方式工作。

以下列出了可用于配置 TCP 预读缓冲功能的 NuttX 配置选项：

* ``CONFIG_NET_TCP_READAHEAD_BUFSIZE``：一个 TCP 预读缓冲区的大小。
* ``CONFIG_NET_NTCP_READAHEAD_BUFFERS``：TCP 预读缓冲区的数量
  （可以为零以禁用 TCP/IP 预读缓冲）

未来的增强是将 TCP 写缓冲区管理逻辑和 TCP 预读缓冲区管理
合并，以便一个公共缓冲区池可用于两种功能
（这可能还需要额外的逻辑来`节流`读缓冲，以便接收到的消息
不会消耗所有缓冲区）。
