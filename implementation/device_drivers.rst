==============
设备驱动
==============

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

标准设备驱动
=======================

设备驱动应在 RTOS 中实现并由应用程序使用。驱动程序为应用程序提供对设备功能的访问。这是模块化 RTOS 设计的必要部分。在 NuttX 目录结构中，可共享的设备驱动位于 ``drivers/`` 下，自定义驱动位于特定板级目录 ``nuttx/boards/<arch>/<chip>/<board>/src`` 或 ``nuttx/boards/<arch>/<chip>/drivers`` 中，这些驱动被构建到 RTOS 中。

总线驱动
===========

在操作系统中有很多东西被称为驱动程序；NuttX 对设备驱动和总线驱动进行了区分。例如，SPI、PCI、PCMCIA、USB、以太网等是总线而非设备。您在 NuttX 架构中永远不会找到总线的设备驱动程序。

在大多数设备架构中，设备驻留在总线上。总线是将驻留在总线上的设备连接到设备驱动程序的传输层。总线由总线驱动程序管理。设备驱动程序使用总线驱动程序传输层的设施与设备进行交互。

以 SPI 为例。SPI 是一种总线。它提供了一个串行总线，可以连接多个设备。SPI 设备驻留在 SPI 总线上，意味着它与 SPI 总线上的其他设备共享相同的 MISO、MOSI 和时钟线（但在 SPI 中，它将拥有自己专用的片选离散线）。

虽然我们通常使用相同的术语"驱动程序"来指代总线驱动程序和设备驱动程序，但有一个重大的根本区别：应用程序只与设备驱动程序交互，从不与总线驱动程序交互。应用程序从不直接与 PCI、PCMCIA、USB、以太网通信，也不与 I2C、SPI 或 GPIO 通信。应用程序通过使用 PCI、PCMCIA、USB、以太网、I2C 或 SPI 的设备驱动程序进行接口。总线驱动程序仅用于支持设备驱动程序与总线上的设备之间的通信。

回到 SPI... 永远不会有一个应用程序可访问的 SPI 接口。如果您的应用程序直接使用 SPI，那么您就将设备驱动程序嵌入到了应用程序中，并违反了 RTOS 的功能分区。

测试驱动
============

可以提供字符驱动程序（如 SPI 驱动程序），代表应用程序执行总线级访问。然而，这种情况在很多情况下是不可接受的。一种可能性是支持总线驱动程序的测试。这里有一个 I2S 的示例：``drivers/audio/i2schar.c``，测试用例在 ``apps/examples/i2schar``。I2S 当然与 SPI 非常相似。此接口仅用于测试目的，可能无法使用它构建任何有意义的应用程序。

I2C 工具
------------

当然，和大多数规则一样，有很多违规情况。I2C 是另一种总线，I2C "驱动程序"是另一种在许多方面与 SPI 类似的传输。对于 I2C，在 ``apps/system/i2c`` 有一个名为"I2C 工具"的应用程序，允许您从命令行访问 I2C 设备。这实际上不仅仅是一个测试工具，也不是应用程序的真正组成部分。

I2C 工具存在一个根本缺陷：它使用 NuttX 内部接口并违反了功能分区。NuttX 有三种构建模式：(1) 平板构建，没有 RTOS 边界的强制执行。在这种平板构建中，I2C 工具工作正常。(2) 内核构建模式和 (3) 保护构建模式。在后两种情况下，操作系统接口是严格强制执行的。在内核和保护构建模式中，I2C 工具不可用，因为它无法访问那些 NuttX 内部接口。

用户空间驱动
==================

上文提到，如果您的应用程序直接使用总线，那么您就将设备驱动程序嵌入到了应用程序中，并违反了 RTOS 的功能分区。在某些上下文中，这种内置于用户应用程序的设备被称为用户空间驱动程序。NuttX 中没有支持用户空间驱动程序的计划或意图。

通信设备
=====================

像 CAN 和 UART 这样的接口怎么办？为什么它们被暴露为驱动程序而 SPI 和 I2C 则不是？

语义是困难的。RTOS 中维护的一般原则是清晰的，但有时在灰色地带中以黑白方式应用原则并不容易。（如果原则妨碍了良好的设计，那么原则应该改变。）

在支持通用设备的真正总线的情况下，这个原则是好的。但也存在灰色地带。

CAN 似乎与以太网相似。两者都是某种网络接口。您不会直接与以太网驱动程序接口，因为您需要通过某种类型的网络协议栈。OSI 模型阻止了这一点。

UART 是通信设备。没有连接设备的 RS-232 总线。相反，总线上有对等方进行通信。这不排除 UART 用作设备驱动程序的低级传输（如无线模块的驱动程序）。也不排除像 Modbus 这样的协议栈层插入路径中。

CAN 与以太网的不同之处在于它确实是直接的点对点通信，更像 UART。尽管您可以在 CAN 上支持像 CANOPen 这样的协议栈。目前 CAN 可以用作简单的字符设备，或使用 SocketCAN 作为网络接口。

通信设备支持基本的点对点模型。CAN 和 UART 基本上是串行接口。但 SPI、I2C 和 USB 也是如此。但后者的串行接口显然与它们相关的主机/设备、主/从模型。将它们视为支持设备接口的总线是非常合理的。

I/O 扩展器
============

I/O 扩展器是一种与 MCU 接口（通常通过 I2C）并提供额外离散输入和输出的设备。同样的规则适用：

* **GPIO 是特定于板级的**。系统中除了特定于板级的逻辑外，不应该有其他组件知道 GPIO。GPIO 可以在板与板之间变化。它们可以出现和消失。它们可以被 GPIO 扩展器取代。您的（可移植的）应用程序不应该有任何关于板上任何离散 I/O 如何实现的知识。NuttX 架构中永远不会有 GPIO 驱动程序。

* **通用驱动程序与板级无关**。通用驱动程序（如 ``drivers/`` 中的驱动程序）也不应该知道任何关于 GPIO 的信息。在所有情况下，板目录中的特定于板级的实现会创建一个"下半部"驱动程序，并将该"下半部"驱动程序与通用"上半部"驱动程序绑定以初始化驱动程序。只有板级逻辑具有任何形式的 GPIO 知识；应用程序和通用"上半部驱动程序"都没有。

* **I2C 和 SPI 驱动程序是内部总线驱动程序**。同样，I2C 和 SPI 驱动程序对应用程序不可访问。这些不是设备驱动程序，而是总线驱动程序。它们不应该被应用程序直接访问。相反，特定于板级的逻辑生成一个"下半部"驱动程序，提供通用的 I2C 或 SPI 接口，并将其与"上半部"驱动程序绑定以初始化驱动程序。

如果您使用 I/O 扩展器，这些规则都不会改变，只是事情变得更加复杂。

示例架构
--------------------

考虑某个 ``<board>`` 的以下情况：

#. 离散操纵杆实现为一组按钮：上、下、左、右和中。每个按钮的状态通过 GPIO 输入感知。

#. GPIO 按钮输入连接到 I2C I/O 扩展器，例如 ``drivers/ioexpander/myexpander.c``，最后连接到

#. 离散操纵杆驱动程序的"上半部"驱动程序（``drivers/input/djoystick.c``）。

实现细节
----------------------

这些应按照以下灵活、可移植、分层的架构实现：

#. 最终，应用程序只通过标准的 open/close/read/ioctl 操作与操纵杆驱动程序接口交互。它将按照 ``include/nuttx/input/djoystick.h`` 中描述的方式接收操纵杆信息。

#. 离散操纵杆驱动程序将在系统初始化时由 ``boards/<arch>/xyz/<board>/src/xyz_djoystick.c`` 等文件中的逻辑初始化。``zyz_joystick.c`` 将创建 ``struct djoy_lowerhalf_s`` "下半部"接口的实例（如 ``nuttx/include/nuttx/input/djoystick.h`` 中所述），并将该接口实例传递给 ``drivers/input/djoystick.c`` "上半部"驱动程序以初始化它。

#. 作为创建 ``struct djoy_lowerhalf_s`` "下半部"接口实例的一部分，``xyz_djoystick.c`` 中的逻辑将执行以下操作：
   它将通过调用 MCU 特定的 I2C 初始化逻辑创建 I2C 驱动程序实例，然后将此 I2C 驱动程序实例传递给 ``drivers/ioexpander/myexpander.c`` 中的 I/O 扩展器初始化接口以创建 I/O 扩展器接口实例。

   请注意，I/O 扩展器接口不应该是普通的字符驱动程序。它不应该通过 open/close/read/write/ioctl 访问。相反，它应该返回某个 ``struct ioexpander_s`` 接口的实例。该 I/O 扩展器接口将在 ``nuttx/include/ioexpander/ioexpander.h`` 中描述。它是内部操作系统接口，永远不会对应用程序逻辑可用。

   收到 I/O 扩展器接口实例后，"下半部"离散操纵杆接口将在内部将其保留为私有数据。系统中除了此"下半部"离散操纵杆驱动程序外，没有任何组件需要知道操纵杆如何在板上连接。

#. 创建"上半部"离散操纵杆接口后，"下半部"离散操纵杆接口将启用来自 I/O 扩展器设备的中断。

#. 当按下按键时，"下半部"离散操纵杆驱动程序将从 I/O 扩展器接收中断。然后它将与 I/O 驱动程序交互以获取当前离散按钮的按下状态。I/O 扩展器驱动程序将与 I2C 交互以获取这些按钮设置。然后将调用离散操纵杆接口回调，为离散操纵杆"上半部"驱动程序提供操纵杆输入。

#. "上半部"离散操纵杆字符驱动程序将响应应用程序代码的 ``read()`` 调用，将编码的操纵杆输入返回给应用程序。
