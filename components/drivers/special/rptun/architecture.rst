============
架构
============

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

RPTUN 位于系统底层，桥接上层 VirtIO/Vhost 驱动与底层 OpenAMP 框架。

概述
========

RPTUN 架构由以下层组成：

::

  +------------------+     +------------------+
  |  VirtIO Drivers  |     |  Vhost Drivers   |
  +--------+---------+     +--------+---------+
           |                        |
           v                        v
  +----------------------------------------+
  |           VirtIO/Vhost Bus             |
  +----------------------------------------+
           |                        |
           v                        v
  +----------------------------------------+
  |              RPTUN Framework           |
  |  +----------------------------------+  |
  |  |       RPTUN Remoteproc           |  |
  |  |  (生命周期管理)                   |  |
  |  +----------------------------------+  |
  |  +----------------------------------+  |
  |  |       VirtIO/Vhost RPTUN         |  |
  |  |  (传输层)                         |  |
  |  +----------------------------------+  |
  +----------------------------------------+
                    |
                    v
  +----------------------------------------+
  |           RPTUN Driver (BSP)           |
  |  (平台特定实现)                        |
  +----------------------------------------+
                    |
                    v
  +----------------------------------------+
  |              OpenAMP                   |
  +----------------------------------------+

组件描述
=====================

RPTUN Remoteproc
----------------

- **功能**：基于 OpenAMP 的 Remoteproc 功能，实现远程 CPU 的生命周期管理（启动、停止、复位等）。它依赖底层驱动 (RPTUN Driver) 提供与启动、关闭和中断相关的芯片特定实现。

- **用户接口**：通过虚拟文件系统 (VFS) 注册字符设备，允许用户空间使用 ``ioctl()`` 系统调用或 NuttShell (NSH) 命令控制远程 CPU。

VirtIO/Vhost RPTUN
------------------

- **功能**：利用 Remoteproc 提供的共享内存和核间中断，实现 VirtIO 标准兼容的 Remoteproc 传输层，使 VirtIO/Vhost 驱动能跨核通信。

- **交互**：RPTUN 通过 ``virtio/vhost_register_device()`` 接口与 VirtIO/Vhost 框架关联。

RPTUN 驱动（适配层）
-------------------------------

- **功能**：实现与生命周期管理、共享内存和核间中断相关的芯片特定功能。

- **职责**：

  - 远程 CPU 启动/停止功能
  - 提供共享内存区域（地址和大小）
  - 实现核间中断发送 (notify) 和接收处理
  - 提供共享内存的物理到设备地址转换表

初始化和启动
==========================

驱动初始化
---------------------

平台驱动调用 ``rptun_initialize()`` 创建并注册 RPTUN 实例。每个实例管理与远程 CPU 的通信通道。通道名称可通过 ``rptun_ops->get_cpuname()`` 接口指定。如果需要与多个远程核通信，必须创建多个 RPTUN 实例。

核心服务启动
--------------------

RPTUN 的核心服务通过 ``rptun_dev_start()`` 启动。此函数创建内核线程执行耗时任务。有两种启动模式：

- **自动启动**：如果驱动配置为自动启动（``RPTUN_IS_AUTOSTART()`` 返回 true），``rptun_initialize()`` 将自动调用 ``rptun_dev_start()``。

- **手动启动**：驱动初始化后，用户可以通过 NSH 命令 ``rptun start /dev/rptun/<cpuname>`` 或 ``ioctl()`` 系统调用手动触发启动过程。

核心初始化任务
-------------------------

在启动线程中，RPTUN 执行以下关键操作：

1. 配置 Remoteproc
2. 获取并解析驱动提供的资源表
3. 根据资源表中的信息创建 VirtIO/Vhost 设备
4. 将创建的设备注册到 VirtIO/Vhost 总线
5. 通过 ``ops->register_callback()`` 注册中断回调函数 ``rptun_callback()`` 以响应来自远程核的中断通知

远程 CPU 生命周期管理
===============================

RPTUN Remoteproc 为用户和内核代码提供两种管理远程 CPU 的方式：

用户空间接口
--------------------

- **NSH 命令**：使用 ``rptun start <dev>`` 和 ``rptun stop <dev>`` 命令
- **字符设备**：打开设备文件 ``/dev/rptun/<cpuname>``，然后使用 ``ioctl()`` 发送 ``RPTUNIOC_START`` 或 ``RPTUNIOC_STOP`` 等控制码

内核空间 API
----------------

内核模块可以直接调用 RPTUN 提供的 API 函数来控制远程 CPU。详见 :doc:`api_reference`。

VirtIO/Vhost 通信流程
===============================

设备初始化
---------------------

在 RPTUN 核心服务启动期间，它解析资源表中定义的每个 VirtIO 设备，创建实际的 VirtIO 设备，并将其注册到 VirtIO/Vhost 总线。主要步骤：

1. **解析资源**：RPTUN 框架解析资源表，查找所有 vdev 和 carveout 资源。

2. **创建设备实例**：根据找到的 vdev 资源信息，调用 OpenAMP 接口 ``remoteproc_create_virtio()`` 创建 VirtIO 设备实例。

3. **初始化共享内存堆**：根据相应的 carveout 资源，初始化独立的共享内存堆。此堆专用于 VirtIO 设备的动态数据缓冲区管理。

4. **注册设备**：将初始化的 VirtIO/Vhost 设备注册到系统，使其对上层应用程序可见。

5. **循环处理**：重复上述步骤，直到资源表中的所有 VirtIO 设备都已创建并注册。

中断处理
------------------

**中断回调（远程到本地）**

当 RPTUN 驱动接收到远程核的核间中断时，它调用注册的 ``rptun_callback()`` 函数。此函数内部调用 ``remoteproc_get_notification()``，最终触发与该 Virtqueue 关联的 VirtIO/Vhost 驱动的回调函数来处理接收到的数据。

**中断通知（本地到远程）**

当 VirtIO/Vhost 驱动需要通知远端（例如，数据已放入 Virtqueue），它调用 ``virtqueue_kick()``。此调用最终调用 RPTUN 框架的 ``rptun_notify()`` 函数，然后调用驱动实现的 ``ops->notify()`` 函数触发硬件发送核间中断。
