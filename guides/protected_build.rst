=====================
NuttX 受保护构建
=====================

.. warning::
    迁移自：
    https://cwiki.apache.org/confluence/display/NUTTX/NuttX+Protected+Build

传统的 "Flat" 构建
============================

传统的 NuttX 构建是 "flat" 构建。所谓 flat，是指当你构建 NuttX 时，
最终得到一个名为 ``nuttx`` 的单一 "blob"。构建的所有组件都位于同一地址空间中。
构建的所有组件都可以访问构建的所有其他组件。

"两遍" 受保护构建
==============================

另一方面，NuttX 受保护构建是 "两遍" 构建，生成两个 "blob"：
(1) 一个单独编译和链接的 `内核` blob，同样称为 `nuttx`，
以及 (2) 一个单独编译和链接的 `用户` blob，在现有构建配置中
称为 ``nuttx_user.elf``。用户 blob 在第一遍创建，
内核 blob 在第二遍创建。

以下两个 make 命令是相同的：

.. code-block:: bash

    make
    make pass1 pass2

但第二个更清晰，我更喜欢将其用于受保护构建。
在第二种情况下，用户和内核 blob 是分别构建的；
在第一种情况下，内核和用户 blob 构建可能会混合在一起，
有些令人困惑。你也可以使用以下命令之一分别构建内核和用户 blob：

.. code-block:: bash

    make pass1
    make pass2

构建结束时，顶层 NuttX 构建目录中将有多个文件。
来自第一遍：

* ``nuttx_user.elf``。第一遍用户空间 ELF 文件
* ``nuttx_user.hex``。第一遍 Intel HEX 格式文件（在 ``defconfig`` 中选择）
* ``User.map``。用户空间 ELF 文件中的符号

来自第二遍：

* ``nuttx``。第二遍内核空间 ELF 文件
* ``nuttx.hex``。第二遍 Intel HEX 文件（在 ``defconfig`` 中选择）
* ``System.map``。内核空间 ELF 文件中的符号

内存保护单元
==========================

如果 MCU 支持内存保护单元（MPU），则内核 blob 中的所有逻辑都以内核模式执行，
即具有所有权限。这些特权线程可以访问所有内存、所有 CPU 指令和所有 MCU 寄存器。
另一方面，在用户模式 blob 中执行的所有逻辑都以用户模式执行，
受到 MCU 和 MPU 强制执行的某些限制。MCU 可能限制对某些寄存器和机器指令的访问；
使用 MPU，用户逻辑对所有内核内存资源的访问都被禁止。
这包括内核 blob 的 FLASH、.bss/.data 存储和内核堆内存。

受保护构建的优势
=================================

这种受保护构建的优势是 (1) 安全性和 (2) 模块化。
由于内核资源受到保护，行为不当的任务使系统崩溃或野指针访问
破坏关键内存的可能性要小得多。这种安全性还提供了更安全的环境
来执行第三方软件，并防止托管应用程序对内核内存的 "窥探"。

模块化是通过严格控制暴露的内核接口来保证的。
在 flat 构建中，所有符号都暴露在外，没有内核 API 的强制执行。
另一方面，在受保护构建中，用户应用程序逻辑与内核的所有交互
必须使用 `系统调用`（或 `syscalls`）与操作系统接口。
系统调用是从用户模式转换到内核模式所必需的；
所有用户空间操作系统接口都通过 syscall `代理`。
然后，在内核模式下，内核系统调用处理程序将执行应用程序请求的操作服务。
在系统处理结束时，恢复用户权限并将控制权返回给用户应用程序。
由于与内核的唯一交互可以通过支持系统调用进行，
因此保证了操作系统的模块化。

用户空间代理/内核空间存根
=====================================

在 "flat" 构建和受保护构建中，相同的操作系统接口暴露给应用程序。
区别在于在受保护构建中，用户代码通过操作系统函数的 `代理` 进行接口。
例如，这是操作系统 ``getpid()`` 接口的代理：

.. code-block:: c

    #include <unistd.h>
    #include <syscall.h>
    pid_t getpid(void)
    {
        return (pid_t)sys_call0(SYS_getpid);
    }

因此 ``getpid()`` 代理是真实操作系统 ``getpid()`` 接口的替代品，
它执行系统调用，以便内核代码可以代表用户应用程序执行真正的 ``getpid()`` 操作。
代理是使用 CSV 文件 ``syscall/syscall.csv`` 和程序 ``tools/mksyscalls``
为所有导出的操作系统接口自动生成的。类似地，在内核端，
有自动生成的 `存根` 将系统调用映射回真正的操作系统调用。
然而，这些对操作系统内部是内部的，实现可能是架构特定的。
有关更多信息，请参阅这些目录中的 ``README.txt`` 文件。

合并 Intel HEX 文件
=========================

你可能面临的一个问题是，两遍构建创建了两个 FLASH 镜像。
我使用的一些调试器允许我将每个镜像分别写入 FLASH。
其他调试器则期望有单个 Intel HEX 镜像。在后一种情况下，
你可能需要将两个 Intel HEX 文件合并为一个。以下是如何操作：

1) ``nuttx.hex`` 文件的 `tail` 应该类似这样（添加了我的注释和空格）：

.. code-block:: bash

    $ tail nuttx.hex
    # 00, data records
    ...
    :10 9DC0 00 01000000000800006400020100001F0004
    :10 9DD0 00 3B005A0078009700B500D400F300110151
    :08 9DE0 00 30014E016D0100008D
    # 05, Start Linear Address Record
    :04 0000 05 0800 0419 D2
    # 01, End Of File record
    :00 0000 01 FF

使用 vi 等编辑器删除 05 和 01 记录。

2) ``nuttx_user.hex`` 文件的 `head` 应该类似这样（同样添加了我的注释和空格）：

.. code-block:: bash

    $ head nuttx_user.hex
    # 04, Extended Linear Address Record
    :02 0000 04 0801 F1
    # 00, data records
    :10 8000 00 BD89 01084C800108C8110208D01102087E
    :10 8010 00 0010 00201C1000201C1000203C16002026
    :10 8020 00 4D80 01085D80010869800108ED83010829
    ...

这里不需要做任何事情。``nuttx_user.hex`` 文件应该没问题。

3) 将编辑过的 nuttx.hex 和未编辑的 ``nuttx_user.hex`` 文件组合
   生成单个组合的 hex 文件：

.. code-block:: bash

    $ cat nuttx.hex nuttx_user.hex >combined.hex

然后将 ``combined.hex`` 文件用于 FLASH/JTAG 工具。
如果你经常这样做，你可能需要花点时间开发一个工具来自动化这些步骤。

文件和目录
=====================

以下是 STM32F4Discovery 受保护构建使用的目录和文件的摘要：

* ``boards/arm/stm32/stm32f4discovery/configs/kostest``。这是内核模式
  OS 测试配置。两个标准配置文件可以在此目录中找到：
  (1) ``defconfig`` 和 (2) ``Make.defs``。
* ``boards/arm/stm32/stm32f4discovery/kernel``。这是第一遍构建目录。
  此目录中的 Makefile 被调用以生成第一遍对象（在此情况下为 ``nuttx_user.elf``）。
  第二遍对象由 ``arch/arm/src/Makefile`` 创建。
  此目录中还有文件 ``userspace.c``。
  用户模式 blob 包含一个头文件，其中包含内核 blob 与用户代码接口所需的信息。
  该头文件由此文件定义。
* ``boards/arm/stm32/stm32f4discovery/scripts``。内核模式构建的链接器脚本
  在此目录中找到。这包括 (1) ``memory.ld``，其中保存公共内存映射，
  (2) ``user-space.ld``，用于链接第一遍用户模式 blob，
  和 (3) ``kernel-space.ld``，用于链接第一遍内核模式 blob。

对齐、区域和子区域
==================================

``memory.ld`` 文件中有一些重要的注释值得在此重复：

"STM32F407VG 有从地址 0x0800:0000 开始的 1024Kb FLASH 和 192Kb SRAM。
SRAM 分为三个块：

* "从地址 0x2000:0000 开始的 112KB SRAM
* "从地址 0x2001:c000 开始的 16KB SRAM
* "从地址 0x1000:0000 开始的 64KB CCM SRAM

"从 FLASH 启动时，FLASH 内存被映射到地址 0x0000:0000，
代码期望通过跳转到 0x0800:0000 地址范围中的入口点开始执行。

"对于 MPU 支持，假设内核模式 NuttX 部分为 128Kb FLASH 和 4Kb SRAM。
这对于应该适合 64KB 的内核来说是过量的，当然可以根据需要进行优化...
允许额外的内存确实允许在内核空间中添加额外的调试工具而不会溢出分区。

"用户空间 FLASH 分区的对齐也是一个关键因素：用户空间 FLASH 分区
将被一个大小为 2||n 字节的单个区域覆盖。
用户空间区域的对齐方式必须相同。因此，随着用户空间大小的增加，
对齐要求也会增加。

"此对齐要求意味着你所能拥有的最大用户空间 FLASH 区域将是 512KB，
它必须位于 0x08800000。如果你更改此地址，不要忘记更改
``CONFIG_NUTTX_USERSPACE`` 配置设置以匹配，并修改 ``kernel/userspace.c`` 中的检查。

"出于同样的原因，SRAM 映射的最大大小限制为 4KB。
这两个对齐限制都可以通过使用多个 MPU 区域来映射 FLASH/SDRAM 范围
或通过巧妙使用子区域来降低。"

内存管理
=================

目前，NuttX 受保护构建有两种内存管理选项：

单一用户堆
----------------

默认情况下，只有一个用户空间堆和堆分配器，
由内核模式和用户模式共享。
优点：简单且充分利用堆内存空间。
缺点：架构尴尬且内核模式分配没有安全性。

双分区堆
-----------------------

两个配置选项可以改变此行为：

* ``CONFIG_MM_MULTIHEAP=y``。这会更改内部内存管理器接口以支持多个堆。
* ``CONFIG_MM_KERNEL_HEAP=y``。使用多堆功能启用内核堆。

如果定义了这两个选项，则构建两个堆分区和两个内存分配器的副本：

一个不受保护的堆分区，将分配内核和用户空间代码共享的用户可访问内存。
该分配器物理位于用户地址空间中，以便可以由用户和内核空间代码直接调用。
用户空间 blob 的开头有一个头文件；内核空间代码从此头文件获取用户空间分配器的地址。

另一个受保护的堆分区，将分配仅可从内核代码访问的受保护内存。
此分配器内置在内核块中。如果你想支持安全功能，则需要此单独的受保护堆。

注意：以内核模式调用用户空间分配器存在安全问题。
这是一个可以被利用来获得系统控制权的安全漏洞！
相反，内核代码应在进入内存分配器存根之前切换到用户模式
（也许通过陷阱）。内存分配器存根随后应陷阱返回到内核模式
（就像信号处理程序现在做的那样）。

传统方法
------------------------

更传统的方法会使用类似 ``sbrk()`` 的接口。
``sbrk()`` 函数向调用进程的堆空间分配添加内存。
在这种情况下，仍然会有内存分配器的内核和用户模式实例。
每个都会根据需要 ``sbrk()`` 以扩展其堆；
为内核模式分配器分配的页面将受到保护，
但为用户模式分配器分配的页面则不会。
优点：满足所有需求。缺点：复杂。由于量化导致的内存损失。

此方法适用于具有非常强大的内存管理单元（MMU）的 CPU，
可以将 sbrk 的块合并为连续的 `虚拟` 堆区域。
没有 MMU，sbrk 的内存将不连续；
这将由于物理页面而限制分配大小。

许多 MCU 将具有内存保护单元（MPU），只能支持安全功能（仅此而已）。
然而，这些低端 MPU 可能不支持足够的映射能力来支持这种传统方法。
例如，ARMv7-M MPU 仅支持八个保护区域来管理所有 FLASH 和 SRAM，
因此这种方法在技术上对 ARMv7-M 系列（Cortex-M3/4）不可行。

比较 "Flat" 构建配置与受保护构建配置
===============================================================================

比较，例如配置 ``boards/arm/stm32/stm32f4discovery/configs/ostest``
和配置 ``boards/arm/stm32/stm32f4discovery/configs/kostest``。
这两个配置是相同的，只是一个构建了 OS 测试的 "flat" 版本，
另一个构建了 OS 测试的内核版本。
有关这些配置的更多详情，请参阅文件
``boards/arm/stm32/stm32f4discovery/README.txt``。

可以使用 ``cmpconfig`` 工具比较配置：

.. code-block:: bash

    cd tools
    make -f Makefile.host cmpconfig
    cd ..
    tools/cmpconfig boards/arm/stm32/stm32f4discovery/configs/ostest/defconfig boards/arm/stm32/stm32f4discovery/configs/kostest/defconfig

以下是所有重要配置差异含义的摘要。
这应该为你提供足够的信息将任何配置从 "flat" 转换为受保护构建：

* ``CONFIG_BUILD_2PASS=y``。启用两遍构建。
* ``CONFIG_BUILD_PROTECTED=y``。此选项启用 "两遍" 受保护构建。
* ``CONFIG_PASS1_BUILDIR="boards/arm/stm32/stm32f4discovery/kernel"``。
  这告诉构建系统第一遍构建目录的（相对）位置。
* ``CONFIG_PASS1_OBJECT=""``。在某些 "两遍" 构建配置中，
  构建系统需要知道第一遍对象的名称。
  此设置不用于受保护构建。
* ``CONFIG_NUTTX_USERSPACE=0x08020000``。这是用户模式 blob 将位于的
  预期位置。用户模式 blob 包含一个头文件，
  其中包含内核 blob 与用户代码接口所需的信息。
  该头文件预计位于此位置。
* ``CONFIG_PASS1_TARGET="all"``。这是用于调用第一遍 make 的构建目标。
* ``CONFIG_MM_MULTIHEAP=y``。这会更改内部内存管理器接口以支持多个堆。
* ``CONFIG_MM_KERNEL_HEAP=y``。NuttX 支持使用单个用户可访问堆的选项，
  或者如果定义了此选项，则支持两个堆：
  (1) 一个将分配内核和用户空间代码共享的用户可访问内存，
  (2) 一个将分配仅可从内核代码访问的受保护内存。
  如果你想支持安全功能，则需要单独的堆内存。
* ``CONFIG_MM_KERNEL_HEAPSIZE=8192``。这确定内核堆的近似大小。
  标准堆空间被分区为内核和用户堆空间。此内核堆大小仅为近似值，
  因为用户堆受到严格的对齐要求。由于对齐要求，
  内核堆的实际大小可能比这大得多。
* ``CONFIG_BOARD_EARLY_INITIALIZE=y``。此设置启用特殊的 `早期` 初始化调用
  以初始化板特定资源。
* ``CONFIG_BOARD_LATE_INITIALIZE=y``。此设置启用特殊的初始化调用
  以初始化 `晚期` 板特定资源。
  ``CONFIG_BOARD_EARLY_INITIALIZE`` 和 ``CONFIG_BOARD_LATE_INITIALIZE``
  之间的区别在于，``CONFIG_BOARD_EARLY_INITIALIZE`` 逻辑在初始化的
  更早阶段运行，在完整操作系统启动并运行之前。
  另一方面，``CONFIG_BOARD_LATE_INITIALIZE`` 在初始化完成时运行，
  就在用户应用程序启动之前。
  ``CONFIG_BOARD_EARLY_INITIALIZE`` 和 ``CONFIG_BOARD_LATE_INITIALIZE``
  都不在 OS 测试配置中使用，但其他配置（如 NSH）
  需要在应用程序运行之前进行一些应用程序特定的初始化。
  在 "flat" 构建中，此类初始化作为应用程序启动序列的一部分执行。
  这包括初始化设备驱动等内容。
  这些相同的初始化步骤必须在受保护构建中以内核模式执行，
  使用 ``CONFIG_BOARD_LATE_INITIALIZE``。
  有关此类板初始化代码的示例，请参阅
  ``boards/arm/stm32/stm32f4discovery/src/up_boot.c``。

架构特定选项：

* ``CONFIG_SYS_RESERVED=8``。用户应用程序逻辑使用系统调用与内核 blob 接口。
  架构特定逻辑可能需要保留一些系统调用供其内部使用。
  ARMv7-M 架构都需要 8 个保留的系统调用。
* ``CONFIG_SYS_NNEST=2``。系统调用可能是嵌套的。
  系统必须保留有关每个嵌套系统调用的信息，
  此设置用于为嵌套系统调用留出资源。
  在当前架构中，最大嵌套级别为 2 就足够了。
* ``CONFIG_ARMV7M_MPU=y``。此设置启用对 ARMv7-M 内存保护单元（MPU）的支持。
  MPU 用于禁止用户模式对内核资源的访问。
* ``CONFIG_ARMV7M_MPU_NREGIONS=8``。ARMv7-M MPU 支持 8 个保护区域。

大小扩展
==============

受保护构建当然会导致 FLASH 镜像比相应的 "flat" 构建更大。
大多少？我手头没有数据，但你可以构建
``boards/arm/stm32/stm32f4discovery/configs/nsh`` 和
``boards/arm/stm32/stm32f4discovery/configs/kostest``
并使用 ``size`` 命令自己比较生成的二进制文件。

大小增加是预期的，因为：

* 受保护构建中包含 syscall 层，但 flat 构建中没有。
* 内核大小的 _syscal_l 存根将导致所有启用的操作系统代码被引入构建。
  在 flat 构建中，只有应用程序实际调用的操作系统接口才会包含在最终对象中。
* 双重内存分配器将增加大小。
* 代码重复。一些代码，如 C 库，将在内核和用户 blob 中重复，以及
* 对齐。MPU 逻辑所需的对齐将留下相对较大的 FLASH（也许 RAM）区域不可用。

性能问题
==================

使用受保护构建的唯一性能差异应该是 `syscalls` 与 flat 构建中
使用的直接 C 调用的交互结果。如果你的性能高度依赖于高速率的操作系统调用，
那么这对你来说可能是个问题。但在典型的应用程序中，
操作系统调用通常不在关键性能路径中。

`syscalls` 最终是软件中断。如果平台不支持优先级、嵌套中断，
那么 `syscall` 执行也可能延迟其他硬件中断处理。
然而，`syscall` 处理是可忽略的：它们实际上只是配置为返回到管理器模式
并转向 `syscall` 存根。它们应该非常快，对于典型的实时应用程序，
应该不会造成任何问题。
