问题
========

NSH 输出奇怪的结果：

.. code-block:: bash

    nsh> sysinfo
    System Information:
        NuttX Version:    0.0 Build: 0
        System Time:    1325809119 [s] UTC
    
    nsh> ps
    PID   PRI SCHD TYPE   NP STATE    NAME
        0   0 FIFO KTHREAD   READY    <noname>()
        1  50 FIFO KTHREAD   WAITSIG  <noname>()
        2 100 FIFO TASK      RUNNING  <noname>()

没有 NAME，也没有版本 / 构建号

答案
======

这可能是正常行为。这里有两个独立的、不相关的问题。

.. _versioning:

版本控制
----------

获取 NuttX 有两种不同的方式：(1) 你可以在 https://bitbucket.org/nuttx/nuttx/downloads 下载版本化的发布，或者你可以 (2) 从 https://github.com/apache/nuttx 的 GIT 仓库获取未版本化的快照。由于你没有版本信息，我假设你使用的是未版本化的副本。

你看到的版本号来自头文件 ``nuttx/include/nuttx/version.h``。该头文件是在构建时从你可以在顶层 nuttx 目录中找到的隐藏文件 .version 创建的。对于 NuttX-7.10，该文件如下所示：

.. code-block:: bash

    #!/bin/bash

    CONFIG_VERSION_STRING="7.10"
    CONFIG_VERSION_MAJOR=7
    CONFIG_VERSION_MINOR=10
    CONFIG_VERSION_BUILD="85981b37acc215ab795ef4ea4045f3e85a49a7af"

``.version`` 文件不存在于 GIT 仓库中；它是在创建 NuttX-7.10 版本时添加到 ``nuttx-7.10.tar.gz`` 压缩包中的。

然后 ``version.h`` 头文件由 ``tools/mkversion`` 在你第一次构建 RTOS 时生成。该工具为上述 ``.version`` 文件生成以下 ``version.h`` 头文件：

.. code-block:: c

    /* version.h -- Autogenerated! Do not edit. */
 
    #ifndef __INCLUDE_NUTTX_VERSION_H
    #define __INCLUDE_NUTTX_VERSION_H
    
    #define CONFIG_VERSION_STRING "7.10"
    #define CONFIG_VERSION_MAJOR 7
    #define CONFIG_VERSION_MINOR 10
    #define CONFIG_VERSION_BUILD "85981b37acc215ab795ef4ea4045f3e85a49a7af"
    
    #define CONFIG_VERSION ((CONFIG_VERSION_MAJOR << 8) | (CONFIG_VERSION_MINOR))
    
    #endif /* __INCLUDE_NUTTX_VERSION_H */

这就是 sysinfo 命令获取并打印的版本信息。

如果你使用的是来自 GIT 仓库的未版本化 NuttX 快照，那么 ``.version`` 文件将不存在。每次构建时，make 系统都会检查是否存在 ``.version`` 文件。如果顶层 nuttx 目录中没有 ``.version``，make 系统将使用 ``tools/version.sh`` 脚本创建一个版本为 0.0 的文件：

.. code-block:: bash

    $(TOPDIR)/.version:
        $(Q) if [ ! -f .version ]; then \
            echo "No .version file found, creating one"; \
            tools/version.sh -v 0.0 -b 0 .version; \
            chmod 755 .version; \
        fi

如果你使用的是未版本化的代码，这完全是适当的。但是，你可以自由编辑顶层 ``.version`` 文件以生成任何你想要的自定义版本信息。然而，说你使用的是发布版本而实际上不是，可能是不合适的。

任务名称大小
--------------

这个很简单。任务名称的大小由 ``.config`` 文件中的以下设置控制：

.. code-block:: c

    CONFIG_TASK_NAME_SIZE=0

它提供任务名称的最大长度。零当然意味着不支持任务名称。
