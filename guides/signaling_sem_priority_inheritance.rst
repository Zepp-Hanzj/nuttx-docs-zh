=============================================
信号发送信号量与优先级继承
=============================================

.. warning:: 迁移自 
    https://cwiki.apache.org/confluence/display/NUTTX/Signaling+Semaphores+and+Priority+Inheritance

锁定与信号发送信号量
===============================

锁定信号量
------------------
POSIX 计数信号量有多种用途。典型用法是信号量用作一个或多个资源的锁。在这种典型情况下，优先级继承工作得非常好：必须记住信号量计数的持有者，以便在更高优先级的任务需要信号量计数时可以提升其优先级。它一直是持有者，直到同一任务调用 ``sem_post()`` 释放信号量上的计数。

互斥示例
------------------------
这种用法非常常见，用于提供互斥。信号量被初始化为值一。第一个获取信号量的任务获得访问权限；需要访问的其他任务将阻塞，直到第一个持有者调用 ``sem_post()`` 放弃访问：

+---------------------+--------------------+
|     **TASK A**      |     **TASK B**     |
+=====================+====================+
| `have access`       |                    |
+---------------------+--------------------+
| `priority boost`    | **sem_wait(sem);** |
+---------------------+--------------------+
| `priority restored` | `have access`      |
+---------------------+--------------------+
| **sem_post(sem);**  |                    |
+---------------------+--------------------+
| **sem_wait(sem);**  |                    |
+---------------------+--------------------+
|                     | `blocked`          |
+---------------------+--------------------+

需要注意的重要事项是 ``sem_wait()`` 和 ``sem_post()`` 都在同一个线程 TASK A 上调用。当 ``sem_wait()`` 成功时，TASK A 成为信号量的持有者，在它是信号量持有者期间 (1) 其他线程（如 TASK B）无法访问受保护的资源，(2) TASK A 的优先级可能被优先级继承逻辑修改。TASK A 一直是持有者，直到它在 `同一线程` 上调用 ``sem_post()``。那时，(1) 它的优先级可能被恢复，(2) TASK B 获得对资源的访问权限。

信号发送信号量
--------------------
但信号量的一个非常不同的使用模型是用于信号发送事件。在这种情况下，信号量计数被初始化为零，接收任务调用 ``sem_wait()`` 等待下一个感兴趣的事件发生。当另一个任务（甚至中断处理程序）检测到感兴趣的事件时，调用 ``sem_post()``，将计数增加到 1 并唤醒接收任务。

信号发送信号量与优先级继承详情
=====================================================

示例
-------
例如，在以下场景中 TASK A 等待信号量上的事件，TASK B（或可能是中断处理程序）通过向该信号量发送来通知任务 A 事件的发生：

+--------------------------+--------------------+
|        **TASK A**        |     **TASK B**     |
+==========================+====================+
| **sem_init(sem, 0, 0);** |                    |
+--------------------------+--------------------+
| **sem_wait(sem);**       |                    |
+--------------------------+--------------------+
| `blocked`                |                    |
+--------------------------+--------------------+
|                          | **sem_post(sem);** |
+--------------------------+--------------------+
| `Awakens as holder`      |                    |
+--------------------------+--------------------+

注意，与上面的互斥情况不同，``sem_wait()`` 和 ``sem_post()`` 是在 `不同的` 线程上调用的。

驱动中的使用
----------------

这种使用情况在驱动中经常使用，例如，当用户调用 ``read()`` 方法且没有可用数据时。调用 ``sem_wait()`` 等待接收新数据；当新数据到达时调用 ``sem_post()``，用户任务被重新唤醒。

优先级继承失败
--------------------------

这两种使用模型——锁定模型和信号发送模型——实际上非常不同，当信号量用于信号发送而非锁定时，优先级继承根本不适用。在这种信号发送情况下，优先级继承可能会干扰信号量的操作。问题在于，当 TASK A 被唤醒时，它是信号量的持有者。通常，当任务最终通过 ``sem_post()`` 释放信号量时，它会从持有者列表中移除。

在这种情况下，TASK B 调用 ``sem_post(sem)`` 但 TASK B 不是信号量的持有者。由于 TASK A 从未调用 ``sem_post(sem)``，它成为信号量的永久持有者，当任何其他任务尝试获取信号量时，其优先级可能随时被提升。

谁该负责
--------------

在 POSIX 的情况下，优先级继承仅在 pthread 互斥层中指定。而在 NuttX 中，pthread 互斥锁是简单地构建在二进制锁定信号量之上的。因此，在 NuttX 中，优先级继承在信号量层中实现。

对于互斥锁的情况，这可以简单地解决，因为只有一个持有者，但对于计数信号量的情况，可能有多个持有者，如果持有者不是调用 ``sem_post()`` 的线程，那么就无法知道应该释放哪个线程/持有者。

选择信号量协议
================================

``sem_setprotocol()``
---------------------

修复方法是在 ``sem_init()`` 之后立即调用非标准 NuttX 函数 ``sem_setprotocol(SEM_PRIO_NONE)``。此函数调用的效果是禁用该特定信号量的优先级继承。因此，用于信号发送的信号量上不应有优先级继承操作。

.. code-block:: C

    sem_t sem
    // ...
    sem_init(&sem, 0, 0);
    sem_setprotocol(&sem, SEM_PRIO_NONE);

规则是：如果你启用了优先级继承并使用信号量进行信号发送，那么你 `必须` 在初始化信号量后立即调用 ``sem_setprotocol(SEM_PRIO_NONE)``。

为什么又一个非标准 OS 接口？
--------------------------------------

非标准的 ``sem_setprotocol()`` 是 POSIX ``pthread_mutexattr_setprotocol()`` 的 `道德` `等价物`，其命名反映了这种关系。在大多数实现中，优先级继承仅在 pthread 互斥层中实现。而在 NuttX 中，pthread 互斥锁是简单地构建在二进制锁定信号量之上的。因此，在 NuttX 中，优先级继承在信号量层中实现。这种架构需要一个像 ``sem_setprotocol()`` 这样的接口来管理底层信号量的协议。


``pthread_mutexattr_setprotocol()``
-----------------------------------

由于 NuttX 在二进制信号量之上实现 pthread 互斥锁，上述建议也适用于 pthread 互斥锁用于线程间信号发送的情况。也就是说，用于信号发送的互斥锁应这样初始化（简化版，此处无错误检查）：

.. code-block:: c

    pthread_mutexattr_t attr;
    pthread_mutex_t mutex;
    // ...
    pthread_mutexattr_init(&attr);
    pthread_mutexattr_settype(&attr, PTHREAD_PRIO_NONE);
    pthread_mutex_init(&mutex, &attr);

这总是问题吗？
=========================

理想情况下，应为所有信号发送信号量调用 ``sem_setprotocol(SEM_PRIO_NONE)``。但是，不，通常信号发送信号量与优先级反转一起使用不是问题。如果信号发送信号量总是在同一线程上获取，则不是问题。例如：

* 如果驱动只被单个任务使用，或者
* 如果信号量只在工作线程上获取。

但如果多个任务可能等待信号发送信号量，这可能是一个严重的问题。像串口驱动这样的驱动，例如，有许多用户线程可能会调用驱动。
