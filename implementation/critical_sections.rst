=================
临界区
=================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

临界区的类型和影响
======================================

临界区是一段简短的代码序列，通过在该代码序列执行期间全局禁用其他活动来保证独占执行。当我们在这里讨论临界区时，我们实际上指的是两种机制之一：

* **临界区本身** 临界区通过调用 ``enter_critical_section()`` 建立；代码序列通过调用 ``leave_critical_section()`` 退出临界区。对于单 CPU 情况，这相当于简单地禁用中断，但在 SMP 情况下更复杂，还涉及自旋锁。

* **禁用抢占** 这是一种相关机制，由于其对系统影响的相似性而被归入此讨论。当禁用抢占（通过 ``sched_lock()``）时，中断保持启用，但不能发生上下文切换；当前任务被锁定到位，在调度器解锁（通过 ``sched_unlock()``）之前不能被挂起。

使用任一机制都会损害实时性能。
临界区对实时性能的影响在 :doc:`/implementation/preemption_latency` 中讨论。
最终结果是为实时响应添加了一定量的**抖动**。

在操作系统中临界区是不可避免的，因此响应时间中存在一定量的"抖动"是可以预期的。重要的是监控临界区到位的最长时间，以便管理该抖动，使响应时间的变化在可接受范围内。

注意：此讨论适用于正常中断处理。此讨论的大部分内容不适用于 :doc:`/guides/zerolatencyinterrupts`。这些中断不会以相同的方式被屏蔽，本页面中讨论的问题都不适用于这些中断。

单 CPU 临界区
============================

操作系统接口
-------------

在我们讨论 SMP 临界区之前，让我们首先回顾可用的内部操作系统接口及其在单 CPU 情况下的作用：

* ``up_irq_save()``（及其伴随函数 ``up_irq_restore()``）。这些简单的接口只是全局启用和禁用中断。这是在单 CPU 情况下建立临界区的最简单方式。正如其他地方讨论的，它确实对实时行为有副作用。

* 但是，不应直接调用 ``up_irq_save()``。相反，应使用包装宏 enter_critical_section()（及其伴随函数 ``leave_critical_section()``）或 ``spin_lock_irqsave()``（和 ``spin_unlock_irqrestore()``）。在单 CPU 情况下，这些宏被定义为简单的 ``up_irq_save()``（或 ``up_irq_save()``）。不应直接调用它们，而应始终通过这些宏间接调用，以便代码也能在 SMP 环境中工作。

* 最后，还有 ``sched_lock()``（和 ``sched_unlock()``）来禁用（和启用）抢占。也就是说，``sched_lock()`` 将锁定您的内核线程并阻止其他任务运行。中断仍然启用，但其他任务不能运行。


使用 sched_lock() 建立临界区——**不要这样做**
----------------------------------------------------

在单 CPU 情况下，``sched_lock()`` 也能很好地建立临界区。毕竟，如果没有其他任务可以在单 CPU 上运行，那么该任务几乎可以独占访问所有资源（前提是这些资源不与中断处理程序共享）。但是，``sched_lock()`` 绝不能用于建立临界区，因为它在 SMP 情况下工作方式不同。在 SMP 情况下，锁定调度器不提供任何形式的资源独占访问。在其他 CPU 上运行的任务仍然可以自由地做任何它们想做的事情。

SMP 临界区
=====================

``up_irq_save()`` 和 ``up_irq_restore()``
------------------------------------------

如前所述，``up_irq_save()`` 和 ``up_irq_restore()`` 永远不应直接调用。这是因为在多 CPU 系统中行为不同。在多 CPU 情况下，这些函数只在本地 CPU 上启用（或禁用）中断。它们对其他 CPU 上的中断没有影响，因此实际上几乎不起作用。当然它们不提供任何意义上的临界区。

``enter_critical_section()`` 和 ``leave_critical_section()``
-------------------------------------------------------------

**自旋锁**

为了建立临界区，我们还需要使用自旋锁。自旋锁只是在一个处理器中执行的循环。如果处理器 A 设置自旋锁 x，那么处理器 B 将不得不等待自旋锁，如下所示：

.. code-block:: C

  while (test_and_set(x))
   {
   }

其中 test and set 是一个原子操作，它设置内存位置的值但也返回其先前的值。这里我们谈论的是内存总线操作方面的原子性：内存位置的测试和设置必须相对于其他总线操作是原子的。需要某种特殊的硬件支持来实现 ``test_and_set()`` 逻辑。

当任务 A 释放锁 x 时，任务 B 将成功获取自旋锁并继续。

**实现**

不深入 ``enter_critical_section()`` 的实现细节，只需说明它 (1) 在本地 CPU 上禁用中断，(2) 使用自旋锁确保跨所有 CPU 的代码序列的独占访问。

请注意，确实创建了临界区：在临界区内，代码确实对受保护的资源具有独占访问权。然而行为实际上非常不同：

* 在单 CPU 情况下，禁用中断会停止来自任何其他任务的所有可能活动。单 CPU 变成单线程且不可中断。
* 在 SMP 情况下，任务继续在其他 CPU 上运行。只有当这些其他任务尝试进入受临界区保护的代码序列时，其他 CPU 上的这些任务才会被停止。它们将在自旋锁上等待。

``spin_lock_irqsave()`` 和 ``spin_unlock_irqrestore()``
--------------------------------------------------------

**通用中断控制器（GIC）**

ARM 提供了一个特殊的可选子系统，称为 MPCore，提供多核支持。MPCore 的一个组件是通用中断控制器或 GIC。GIC 支持 16 个处理器间中断，是在这些平台上实现 SMP 的关键组件。它们被称为软件生成中断或 SGI。

GIC 的一个奇怪行为是 SGI 不能被禁用（至少不能使用标准的 ARM 全局中断禁用逻辑）。因此禁用本地中断不会阻止这些 GIC 中断。

这在建立临界区时造成了许多复杂性和显著开销。

**ARMv7-M NVIC**

GIC 在所有最近的 ARM 架构中都可用。然而，大多数嵌入式 ARM7-M 多核 CPU 只是将处理器间中断作为可通过 NVIC 屏蔽的正常中断（每个 CPU 将有自己的 NVIC）。

这意味着在这些情况下，临界区逻辑可以大大简化。

**实现**

对于 GIC 不支持禁用中断的情况，
``spin_lock_irqsave()`` 和 ``spin_unlock_irqstore()`` 等同于
``enter_critical_section()`` 和 ``leave_critical_section()``。只有在处理器间中断可以被禁用的情况下才存在差异。

在那种情况下，``spin_lock_irqsave()`` 将禁用本地中断并获取自旋锁。这是一个非常简单高效的临界区实现。

然而，有两个重要的注意事项：

* 此临界区内的逻辑绝不能挂起！例如，如果代码调用 ``spin_lock_irqsave()`` 然后 ``sleep()``，那么睡眠将在自旋锁处于锁定状态时发生，整个系统可能会被阻塞。相反，``spin_lock_irqsave()`` 只能与直线代码一起使用。

* 这与通过 ``enter_critical_section()`` 建立的临界区不同。获取一个临界区不会阻止另一个 CPU 上的逻辑获取另一个临界区，结果是您可能没有您认为拥有的保护。

``sched_lock()`` 和 ``sched_unlock()``
---------------------------------------

除了一些细节外，SMP 的 ``sched_lock()`` 的工作方式与单 CPU 情况非常相似。以下是注意事项：

* 与单 CPU 情况一样，调用 ``sched_lock()`` 的情况被锁定到位，不能被挂起。

* 然而，任务将继续在其他 CPU 上运行，因此 ``sched_lock()`` 不能用作临界区。

* 其他 CPU 上的任务也被锁定到位。但是，它们可以选择在任何时候挂起自己（例如通过 ``sleep()``）。在这种情况下，只允许运行该 CPU 的 IDLE 任务。

临界区监视器
============================

内部操作系统钩子
-----------------

**临界区监视器**

为了测量任务持有临界区的时间，操作系统支持临界区监视器。这是内部检测工具，记录任务持有临界区的时间。它还记录全局禁用中断的时间量。临界区监视器然后保留临界区到位的最长时间，包括每任务和全局。我们还扩展了临界区监视器以进行任务调度成本统计，这可以高效地进行 CPU 负载统计。为了在您只关注特定功能时节省不必要的开销，我们将 critmon 功能隔离到不同的配置中。允许您只打开某些功能以最小化对性能等方面的副作用。

临界区监视器通过配置中的以下设置启用::

  CONFIG_SCHED_CRITMONITOR=y

全局启用调度 critmon，所有其他功能需要此配置作为前缀。

**线程执行**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_THREAD=0

* 默认 0 启用执行时间统计，并使其成为支持 cpuload 的来源。
* > 0 还在执行时间超过配置的滴答数时执行警报日志。
* -1 禁用线程执行时间统计功能。

如果您在 critmon 中没有更多需求，此方法**推荐**作为 cpuload 后端。当禁用 critmon 中的所有其他统计时，此方法是一种高效的方式进行 CPU 负载统计。因为我们没有在临界区和抢占操作中添加钩子。只有在调度器触发上下文切换时才有指令。

**工作队列执行**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_WQUEUE=-1

* 默认 -1 禁用工作队列最大执行时间
* > 0 在工作队列执行时间超过配置的滴答数时执行警报日志。

**禁用抢占的时间**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_PREEMPTION=-1

* 默认 -1 禁用禁用抢占时间统计。
* >= 0 启用禁用抢占时间统计，数据将在 critmon procfs 中。
* > 0 还在禁用抢占时间超过配置的滴答数时执行警报日志。

**进入临界区的时间**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_CSECTION=-1

* 默认 -1 禁用进入临界区时间统计。
* >= 0 启用进入临界区时间统计，数据将在 critmon procfs 中。
* > 0 还在进入临界区时间超过配置的滴答数时执行警报日志。

**临界区进入前忙等待时间**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_BUSYWAIT=-1

* 默认 -1 禁用临界区进入前忙等待时间统计。
* >= 0 启用临界区进入前忙等待时间统计，数据将在 critmon procfs 中。
* > 0 还在临界区进入前忙等待时间超过配置的滴答数时执行警报日志。

**IRQ 执行时间**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_IRQ=-1

* 默认 -1 禁用 IRQ 执行时间统计。
* >= 0 启用 IRQ 执行时间统计，数据将在 critmon procfs 中。
* > 0 还在 IRQ 执行时间超过配置的滴答数时执行警报日志。

**看门狗执行时间**::

  CONFIG_SCHED_CRITMONITOR_MAXTIME_WDOG=-1

* 默认 -1 禁用看门狗执行时间统计。
* >= 0 启用看门狗执行时间统计，数据将在 critmon procfs 中。
* > 0 还在看门狗执行时间超过配置的滴答数时执行警报日志。

**每线程和全局临界区**

在 NuttX 中，临界区在每任务的基础上控制。例如，考虑以下代码序列：

.. code-block:: C

   irqstate_t flags = enter_critical_section();
   sleep(5);
   leave_critical_section(flags);

任务，比如任务 A，通过 ``enter_critical_section()`` 建立临界区，但当任务 A 被 ``sleep(5)`` 语句挂起时，它放弃了临界区。系统状态将由下一个要恢复的任务决定，比如任务 B：通常，下一个任务不会在临界区内，因此当任务睡眠时临界区被打破。当任务 A 在睡眠时间到期后再次运行时，该临界区将被重新建立。

但是，如果恢复的任务 B 也在临界区内，那么临界区将被延长更长时间！这就是为什么临界区到位的全局时间可能比任何单个线程持有临界区的时间都长。

ProcFS
------

操作系统通过 ProcFS 文件系统报告这些最长时间，通常挂载在 ``/proc``：

* ``/proc/<ID>/critmon`` 伪文件报告线程 ID = <ID> 的每线程最大值。系统中每个活动任务都有一个此 critmon 文件的实例。

* ``/proc/critmon`` 伪文件报告 CPU 全局状态的类似信息。

``/proc/<ID>/critmon`` 文件的输出格式为::

  X.XXXXXXXXX,X.XXXXXXXXX

其中 ``X.XXXXXXXXX`` 是以秒为单位的时间，具有纳秒精度（但不一定是准确度，准确度取决于时钟源）。第一个数字是持有禁用抢占的最长时间；第二个数字是持有临界区的最长时间。

此文件可以从 NSH 读取，如下所示：

.. code-block:: bash

   nsh> cat /proc/1/critmon
   0.000009610,0.000001165

``/proc/critmon`` 文件的输出格式类似::

  X,X.XXXXXXXXX,X.XXXXXXXXX

其中第一个 X 是 CPU 编号，后面的两个数字与 ``/proc/<ID>/critmon`` 的解释相同。在单 CPU 情况下，伪文件中将有一行，``X=0``；在 SMP 情况下，将有多行，每个 CPU 一行。

此文件也可以从 NSH 读取：

.. code-block:: bash

   nsh> cat /proc/critmon
   0,0.000009902,0.000023590

每次读取伪文件时都会清除这些统计信息，因此报告的值是自上次读取 ProcFS 伪文件以来的最大值。

``apps/system/critmon``
-----------------------

还有一个位于 ``apps/system/critmon`` 的应用程序守护程序。此守护程序定期读取上面描述的 ProcFS 文件并将输出转储到 stdout。此守护程序通过以下方式启用：

.. code-block:: bash

   nsh> critmon_start
   Csection Monitor: Started: 3
   Csection Monitor: Running: 3
   nsh>
   PRE-EMPTION CSECTION    PID   DESCRIPTION
   MAX DISABLE MAX TIME
   0.000100767 0.000005242  ---  CPU 0
   0.000000292 0.000023590     0 Idle Task
   0.000036696 0.000004078     1 init
   0.000000000 0.000014562     3 Csection Monitor
   ...

并通过以下方式停止：

.. code-block:: bash

   nsh> critmon_stop
   Csection Monitor: Stopping: 3
   Csection Monitor: Stopped: 3

IRQ 监视器和最坏情况响应时间
========================================

IRQ 监视器是额外的操作系统检测工具。IRQ 监视器的完整讨论超出了本页面的范围。只需说明：

* IRQ 监视器通过 ``CONFIG_SCHED_IRQMONITOR=y`` 启用。

* IRQ 监视器收集的数据在 ``/proc/irqs`` 中提供。

* 此数据也可以使用 ``nsh> irqinfo`` 命令查看。

* 此数据包括每个 IRQ 接收的中断数以及处理中断所需的时间，从进入附加的中断处理程序直到退出中断处理程序。

根据这些信息，我们可以计算从中断请求到可以处理中断的任务运行的最坏情况响应时间。
该最坏情况响应时间 ``Tresp`` 由以下给出：

* ``Tresp1 = Tcrit + Tintr + C1``

* ``Tresp2 = Tintr + Tpreempt + C2``

* ``Tresp = MAX(Tresp1, Tresp2)``

其中：

* ``C1`` 和 ``C2`` 是未知的、不可约的常数，反映硬件中断延迟和上下文切换时间等因素，

* ``Tcrit`` 是在临界区内观察到的最长时间，

* ``Tintr`` 是感兴趣的事件的中断处理程序执行所需的时间，以及

* ``Tpreempt`` 是禁用抢占时观察到的最长时间。

注意：

#. 此计算假设感兴趣的任务是系统中最高优先级的任务。它不考虑响应任务因优先级不足而被延迟的可能性。

#. 此计算未解决干扰任务同时禁用抢占和持有临界区的情况。当然 Tresp1 在此情况下是有效的，但 Tresp2 不是。在中断之后和响应任务可以运行之前可能存在一些额外的、未测量的延迟，这取决于临界区释放和抢占重新启用的顺序：

     * 当任务离开临界区时，无论是否启用抢占，挂起的中断都将立即执行。

     * 如果首先启用抢占，那么中断后不会有延迟，因为当中断返回时抢占将被启用。

     * 如果任务首先离开临界区，那么在中断返回之后和响应任务可以运行之前会有一些小的未知持续时间的延迟，因为当中断返回时抢占将被禁用。

#. 此计算未解决并发中断。所有中断都在相同的优先级运行，如果在中断处理程序内发生中断请求，那么它必须挂起直到该中断完成。因此，也许上面的 ``Tresp1`` 公式应该是以下？（这假设硬件仲裁使得感兴趣的中断最多被一个中断延迟）。并发、嵌套中断可能通过优先级化得到更好的支持。参见更多：:doc:`/guides/nestedinterrupts`。

     * ``Tresp1 = Tcrit + Tintrmax + Tintr + C1``

       其中：

       * ``Tintrmax`` 是所有中断源中最长的中断处理时间（不包括正在考虑的事件的中断）。

您能做什么？
----------------

如果时序数据表明您无法满足截止时间，您能做什么？您有以下选择：

#. 使用这些工具找到持有临界区或禁用抢占时间过长的确切函数。然后优化该函数，使其更快地释放该资源。通常临界区是在长代码序列上建立的，而它们可以被重新设计为在更短的代码序列上使用临界区。

#. 在某些情况下，使用临界区或禁用抢占可以被替换为锁定信号量。使用此类锁的锁定效果的范围不是全局的，而仅限于共享相同资源的任务。临界区应仅用于保护任务级逻辑和中断级逻辑之间共享的资源。

#. 切换到 :doc:`/guides/zerolatencyinterrupts`。这些中断不受本页面讨论的大多数问题的影响。

**注意**

操作系统中有一些地方通过 ``sched_lock()`` 禁用抢占以建立临界区。这是对 ``sched_lock()`` 的不正确使用。``sched_lock()`` 只是防止当前执行的任务被挂起。对于单 CPU 平台的情况，这确实有效地创建了临界区：由于没有其他任务可以运行，锁定任务确实对不与中断级逻辑共享的所有资源具有独占访问权。

但在多 CPU SMP 情况下并非如此。``sched_lock()`` 仍然阻止在 CPU 上运行的当前任务被挂起，但它不支持访问中的任何排他性，因为将有其他任务在其他 CPU 上运行，可能会访问相同的资源。
