====================
工作队列死锁
====================

.. note:: 本文档翻译自 NuttX 官方文档，如需查阅最新版本请访问 https://nuttx.apache.org/docs/latest/

工作队列的使用
==================

大多数网络驱动使用工作队列来处理网络事件。这样做有两个原因：
(1) 大多数可参考的示例代码都这样做，(2) 使用工作队列比创建专用任务/线程
来服务网络更容易且更有效地利用内存资源。

高优先级和低优先级工作队列
=================================

有两个工作队列：一个单一的高优先级工作队列，仅用于在半正常的任务上下文中
服务后端中断处理。以及低优先级工作队列，类似于前者，但如名称所示，
优先级较低，不专门用于时间关键的后端中断处理。

工作队列的缺点
========================

使用工作队列有两个重要的缺点。首先，工作队列本质上是不确定的。
从您调度工作到工作被执行的时间延迟是高度随机的，该延迟不仅由于
严格的优先级调度，还由于排在您前面的工作。

如果您依赖不确定的工作队列来完成大部分工作，为什么还要费心使用 RTOS？

第二个问题是相关的：一次只能执行一个工作队列作业。该作业应该简短，
以便尽快将工作队列释放给下一个工作队列作业。该作业绝不应该阻塞等待资源！
如果作业阻塞，它会阻塞整个工作队列，并使整个工作队列在等待期间不可用。

工作队列上的网络
=========================

如前所述，大多数网络驱动使用工作队列来处理网络事件。
（有些甚至可配置为使用高优先级工作队列……太可怕了！）
大多数网络操作实际上不适合在工作队列上执行：
网络操作可能相当长，也可能阻塞等待资源可用。
因此，至少，网络不应该使用高优先级工作队列。

死锁
=========

如果只有一个工作队列实例，那么当工作作业在工作队列上阻塞时，
很容易在工作队列上造成死锁。以下是通用的工作队列死锁场景：

* 一个作业在工作队列上运行，等待资源可用。
* 提供该资源的操作也在同一工作队列上运行。
* 但由于工作队列正在阻塞等待资源，提供资源的作业无法运行，
  从而导致死锁。

IOB
====

IOB（I/O 块）是小型 I/O 缓冲区，可以链接在一起以高效地缓冲
可变大小的网络数据包数据。这比全尺寸数据包缓冲区更有效地利用缓冲空间，
因为数据包内容通常远小于全尺寸数据包大小（MSS）。

网络分配 IOB 以支持 TCP 和 UDP 预读缓冲和写缓冲。
当接收到 TCP/UDP 数据且没有接收方在等待接受数据时，使用预读缓冲。
在这种情况下，接收到的有效负载被缓冲在基于 IOB 的预读缓冲区中。
当应用程序下次调用 ``recv()`` 或 ``recvfrom()`` 时，
数据将从预读缓冲区中移除并立即返回给调用方。

写缓冲是指传出方向上的类似功能。当应用程序调用 ``send()`` 或 ``sendto()``
且驱动不可用以接受新数据包数据时，数据被缓冲在写缓冲链中的 IOB 中。
当网络驱动最终可用以获取更多数据时，数据包数据从写缓冲中移除并提供给驱动。

IOB 以固定大小分配。系统启动时预分配固定数量的 IOB。
如果网络耗尽 IOB，不会动态分配额外的 IOB，
相反，IOB 分配器 ``iob_alloc()`` 将阻塞等待直到 IOB 最终返回到空闲 IOB 池。
还有一个非阻塞 IOB 分配器 ``iob_tryalloc()``。

在高利用率条件下，如以高速率发送大量数据或以高速率接收大量数据，
系统不可避免地会耗尽预分配的 IOB。对于预读缓冲，在这种情况下数据包被直接丢弃。
对于 TCP，这意味着远程对端随后会超时，因为不会收到 ACK，
远程对端最终会重新传输数据包。UDP 是有损传输，丢失或丢弃的数据报
必须包含在任何 UDP 设计中。

对于写缓冲，当 IOB 池耗尽时可能发生三种行为：
首先，如果在 ``send()`` 或 ``sendto`` 传输开始时没有可用的 IOB，
如果未选择 ``O_NONBLOCK``，操作将阻塞直到 IOB 再次可用。
此延迟可能是相当长的时间。

其次，如果选择了 ``O_NONBLOCK``，发送当然会立即返回，
如果我们无法为传输分配第一个 IOB，将失败并将 errno 设置为 ``EAGAIN``。

第三种行为发生在传输中间 IOB 耗尽时。
然后发送操作不会等待，而是发送已成功缓冲的字节数。
应用程序应始终检查 ``send()`` 或 ``sendto()`` 的返回值。
如果它是小于请求传输大小的字节数，则应再次调用发送函数。

阻塞的 iob_alloc() 调用也是工作队列死锁的常见原因。
场景再次是：

* 操作系统中的某些逻辑在工作队列上运行，阻塞等待 IOB 可用，
* 释放 IOB 的逻辑也在同一工作队列上运行，但
* 提供 IOB 的逻辑无法执行，因为另一个作业在同一工作队列上阻塞等待 IOB。

工作队列的替代方案
===========================

为避免网络死锁，规则是：永远不要在单例工作队列上运行网络！

大多数网络实现就是这样做的！以下是一些替代方案：

#. 使用多个低优先级工作队署
   与高优先级工作队列不同，低优先级工作队列利用线程池。
   线程池中的线程数量由 ``CONFIG_SCHED_LPNTHREADS`` 控制。
   如果 ``CONFIG_SCHED_LPNTHREADS`` 大于一，则此类死锁不应可能：
   在这种情况下，如果一个线程正忙于其他作业（即使它只是在等待资源），
   则该作业将被分配给不同的线程，死锁将被打破。
   额外低优先级工作队列线程的成本主要是为线程栈预留的内存。

#. 使用专用网络线程
   最好的解决方案是编写自定义内核线程来处理驱动网络操作。
   这将是最高性能和最可管理的。然而，它也将需要大量工作。

#. 与网络锁的交互
   网络锁是一个可重入互斥锁，强制对网络进行互斥访问。
   网络锁也可能导致死锁，并且可能与工作队列交互以降低性能。
   考虑以下场景：

     * 某些网络逻辑（可能在应用程序线程上运行）获取网络锁，
       然后等待 IOB 可用（在应用程序线程上，不是工作队列）。
     * 某些网络相关事件在工作队列上运行，但阻塞等待网络锁。
     * 另一个作业排在网络作业之后。这是提供 IOB 的作业，
       但它无法运行，因为另一个线程在工作队列上阻塞等待网络锁。

   但网络不会被解锁，因为应用程序逻辑持有网络锁并正在等待 IOB，
   而 IOB 永远不会被释放。

   在网络内部，此死锁条件通过特殊函数 ``net_ioballoc()`` 避免。
   ``net_ioballoc()`` 是阻塞 ``iob_alloc()`` 的包装器，
   在等待 IOB 可用时暂时释放网络锁。

   类似地，网络函数 ``net_lockedait()`` 和 ``net_timedait()`` 分别是
   ``nxsem_wait()`` 和 ``nxsem_timedwait()`` 的包装器，
   并且在等待期间也释放网络锁。

   使用这些包装器函数时应谨慎。因为在等待期间网络锁被放弃，
   在锁恢复之前可能发生网络状态变化。您的设计应考虑到这种可能性。
