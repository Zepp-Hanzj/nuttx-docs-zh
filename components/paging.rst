================
On-Demand Paging
================

Kernel Build Implementation
===========================

按需分页和延迟加载是用于管理物理内存的技术。基本思想是允许程序即使整个程序不常驻在内存中也能执行。程序根据需要加载到内存中。这是一种在许多操作系统中使用的技术，允许大型程序在小型内存系统上执行。通常，使用内存管理单元 (MMU) 将虚拟内存映射到物理内存。然后将应用程序加载到虚拟内存地址空间中，对物理内存的访问由 MMU 管理。如果虚拟内存不在物理内存中，则会发生缺页异常。然后操作系统将缺失的页面加载到内存中并恢复执行。

Requirements and Assumptions
-----------------------------

按需分页需要 *内核构建* (``CONFIG_BUILD_KERNEL=y``) 模式。在此模式下，不会在 NuttX 内核内构建应用程序。相反，应用程序作为单独的程序构建并加载到内存中 (``CONFIG_ELF=y`` 和 ``CONFIG_BINFMT_LOADABLE=y``)。在此模式下，每个进程有自己的地址环境 (``CONFIG_ARCH_ADDRENV=y``)。

Logic Design Description
-------------------------

当应用程序被加载时，会调用 ``up_addrenv_create`` 来创建进程的地址环境。这包括在虚拟内存空间中映射常用的 ``text``、``data`` 和 ``heap`` 段。如果没有按需分页，则在进程启动之前分配和映射物理内存。当启用按需分页时，通常只为每个段分配和映射一个页面。

进程在其地址环境中开始执行，访问虚拟内存。每当它尝试访问未在 MMU 中映射的虚拟内存地址时，就会发生缺页异常。MMU 触发由内核处理的异常。内核然后检查是否有足够的可用物理页面，并将虚拟内存地址映射到该页面。最后，从缺页首次发生的位置恢复执行。

Example: RISC-V
^^^^^^^^^^^^^^^

RISC-V 的 ``up_addrenv_create`` 调用 ``create_region``（两者都定义在 ``arch/risc-v/src/common/riscv_addrenv.c`` 中）。``create_region`` 通过为页表分配物理内存将单个区域映射到 MMU。当未选择 ``CONFIG_PAGING=y`` 时，所有物理页表从物理内存空间分配并映射到虚拟内存空间。当选择 ``CONFIG_PAGING=y`` 时，只将每个段的第一页映射到虚拟内存空间。其余页面仅在发生缺页时才映射到虚拟内存空间。

缺页由异常处理器中的 ``riscv_fillpage`` 函数处理（定义在 ``arch/risc-v/src/common/riscv_exception.c`` 中）。每当发生缺页时，会调用 ``riscv_fillpage`` 函数。此函数分配一个物理页面，将其映射到触发缺页异常的虚拟内存空间，然后从缺页首次发生的位置恢复执行。

:ref:`knsh_paging` 模拟了一个具有 4MiB 物理内存的设备，每个进程分配 8MiB 的虚拟堆内存。这通过启用按需分页来实现。

Legacy Implementation
=====================

此旧版实现在 *Flat Build* 上运行（当时 *Kernel Build* 尚不存在）。

什么样的平台可以支持 NuttX 旧版按需分页？

  #. MCU 应该有一些大的、可能低成本的非易失性存储，如串行 FLASH 或 SD 卡。此存储可能不支持随机访问（否则，为什么不直接在存储介质上执行程序）。SD 和串行 FLASH 成本低，不需要很多引脚，SPI 支持在几乎所有 MCU 中都很普遍。这个大的串行 FLASH 将包含一个大程序。可能是一个几兆字节大小的程序。
  #. MCU 必须有一个（相对）小的快速 SRAM 块，可以从中执行代码。大小例如 256K（或如 NXP LPC3131 中的 192K）对许多应用来说就足够了。
  #. MCU 有一个 MMU（同样像 NXP LPC3131 一样）。

如果平台满足这些要求，那么 NuttX 可以提供按需分页：它可以根据需要将非易失性介质中的大程序的 .text 复制到 RAM 中，以便从小 RAM 执行大型程序。

Terminology
-----------

  ``g_waitingforfill``：
     一个操作系统列表，用于保存等待页面填充的任务的 TCB。
  ``g_pftcb``：
     一个变量，保存对当前正在被重新填充的线程的 TCB 的引用。
  ``g_pgworker``：
     将执行页面填充的线程的 *进程* ID。
  ``pg_callback()``：
     当填充完成时从驱动调用的回调函数。
  ``pg_miss()``：
     从架构特定代码调用以处理缺页的函数。
  ``TCB``：
     任务控制块

NuttX Common Logic Design Description
-------------------------------------

Initialization
^^^^^^^^^^^^^^

将添加以下声明。

-  ``g_waitingforfill``。一个双向链表，用于实现等待页面填充的任务的 TCB 的优先级列表。
-  ``g_pgworker``。将执行页面填充的线程的 *进程* ID。

在 ``sched/init/nx_start.c`` 的操作系统初始化期间，将执行以下步骤：

-  初始化 ``g_waitingforfill`` 队列。
-  启动特殊的页面填充工作线程。页面填充工作线程的 ``pid`` 将保存在 ``g_pgworker`` 中。注意，我们需要一个特殊的工作线程来执行填充；我们不能使用"通用"工作线程设施，因为我们无法保证该工作线程调用的所有操作都将始终驻留在内存中。

``g_waitingforfill``、``g_pgworker`` 和其他内部私有定义的声明将在 ``sched/paging/paging.h`` 中提供。架构特定代码应使用的所有公共定义将在 ``include/nuttx/page.h`` 中可用。大多数架构特定函数在 ``include/nuttx/arch.h`` 中声明，但对于此分页逻辑，这些架构特定函数改为在 ``include/nuttx/page.h`` 中声明。

Page Faults
^^^^^^^^^^^

**缺页异常处理**。缺页处理由函数 ``pg_miss()`` 执行。此函数从架构特定的内存段错误处理逻辑中调用。此函数将执行以下操作：

#. **健全性检查**。如果当前执行的任务是页面填充工作线程，此函数将 ASSERT。页面填充工作线程是解决缺页的方式，与页面填充工作线程关联的所有逻辑必须被"锁定在内存中"并始终存在于内存中。
#. **阻塞当前执行的任务**。此函数将调用 ``up_switch_context()`` 来阻塞就绪列表头部的任务。这应导致中断级上下文切换到下一个最高优先级的任务。被阻塞的任务将被标记为 ``TSTATE_WAIT_PAGEFILL`` 状态，并将保留在 ``g_waitingforfill`` 优先级任务列表中。
#. **提升页面填充工作线程优先级**。检查 ``g_waitingforfill`` 列表头部任务的优先级。如果该任务的优先级高于页面填充工作线程的当前优先级，则将页面填充工作线程的优先级提升到该优先级。因此，页面填充工作线程将始终以等待填充的最高优先级任务的优先级运行。
#. **通知页面填充工作线程**。是否已有页面正在被填充？如果没有，则通知页面填充工作线程开始处理排队的页面填充请求。

当从 ``pg_miss()`` 收到信号时，页面填充工作线程将被唤醒并启动填充操作。

**输入参数**。无 -- 假定就绪列表的头部是导致异常的任务。当前任务上下文应已保存在该任务的 TCB 中。不需要其他输入。

**假设**。

-  假定此函数从异常处理器级别调用，且所有中断都被禁用。
-  ``pg_miss()`` 必须"锁定在内存中"。调用 ``pg_miss()`` 不会导致嵌套缺页。
-  假定当前执行的任务（就绪列表头部的那个）是导致缺页的那个。除非缺页发生在中断处理器中，否则这将始终为真。中断处理逻辑必须始终可用并"锁定在内存中"，这样缺页永远不会来自中断处理。
-  架构特定的缺页异常处理已验证异常未发生在中断/异常处理逻辑中。
-  如上所述，导致缺页的任务不能是页面填充工作线程，因为那是完成页面填充的唯一方式。

Fill Initiation
^^^^^^^^^^^^^^^

页面填充工作线程将在以下三种条件下被唤醒：

-  当从 ``pg_miss()`` 收到信号时，页面填充工作线程将被唤醒（见上文），
-  从 ``pg_callback()`` 在完成上次填充后（当定义了 ``CONFIG_PAGING_BLOCKINGFILL`` 时...见下文），或
-  可配置的超时到期且无活动。此超时可用于检测失败情况，例如填充永远不完成。

页面填充工作线程将维护一个名为 ``struct tcb_s *g_pftcb`` 的静态变量。如果没有填充正在进行，``g_pftcb`` 将为 NULL。否则，它将指向正在接收填充的任务的 TCB。

当从 ``pg_miss()`` 唤醒时，没有填充正在进行，``g_pftcb`` 将为 NULL。在这种情况下，页面填充工作线程将调用 ``pg_startfill()``。该函数将执行以下操作：

-  调用架构特定函数 ``up_checkmapping()`` 以查看是否仍需要执行页面填充。在某些情况下，缺页可能在多个线程上发生并被多次排队。在这种极端情况下，被阻塞的任务将简单地重新启动（见下面正常完成填充操作的情况的逻辑）。
-  调用 ``up_allocpage(tcb, &vpage)``。此架构特定函数将在内存中预留页面并映射到虚拟地址 (vpage)。如果所有可用页面都在使用中（典型情况），此函数将选择一个正在使用的页面，取消映射，并使其可用。
-  调用架构特定函数 ``up_fillpage()``。支持两个版本的 up_fillpage 函数 -- 基于配置设置 ``CONFIG_PAGING_BLOCKINGFILL`` 的阻塞版本和非阻塞版本。

   -  如果定义了 ``CONFIG_PAGING_BLOCKINGFILL``，则 up_fillpage 是阻塞调用。在这种情况下，``up_fillpage()`` 将只接受 (1) 需要填充的 TCB 的引用。TCB 中的架构特定上下文信息将足以执行填充。以及 (2) 要填充的已分配页面的（虚拟）地址。填充的结果状态将由 ``up_fillpage()`` 的返回值提供。
   -  如果定义了 ``CONFIG_PAGING_BLOCKINGFILL``，则 up_fillpage 是非阻塞调用。在这种情况下 ``up_fillpage()`` 将接受一个额外的参数：页面填充工作线程将提供一个回调函数 ``pg_callback``。此函数是非阻塞的，它将启动异步页面填充。调用非阻塞的 ``up_fillpage()`` 后，页面填充工作线程将等待被通知下一个事件 -- 填充完成事件。当页面填充完成（或发生错误）时将调用回调函数。填充的结果状态将作为参数提供给回调函数。此回调可能从中断级别发生。

在任何情况下，当填充正在进行时，其他任务可以执行。如果在此期间发生另一次缺页，出错的任务将被阻塞，其 TCB 将（按优先级顺序）添加到 ``g_waitingforfill``，并且页面工作线程的优先级可能会提升。但在当前页面填充完成之前不会采取任何操作。注意：IDLE 任务也必须完全"锁定在内存中"。IDLE 任务不能被阻塞。在所有任务都阻塞等待页面填充的情况下，IDLE 任务必须仍然可用以运行。

架构特定函数 ``up_checkmapping()``、``up_allocpage(tcb, &vpage)`` 和 ``up_fillpage(page, pg_callback)`` 将在 ``include/nuttx/arch.h`` 中原型化。

Fill Complete
^^^^^^^^^^^^^

对于阻塞的 ``up_fillpage()``，填充的结果将直接从 ``up_fillpage`` 调用返回。

对于非阻塞的 ``up_fillpage()``，当填充完成时，架构特定驱动调用提供给 ``up_fillpage()`` 的 ``pg_callback()``。在这种情况下，``pg_callback()`` 可能从驱动中断级逻辑调用。驱动将提供填充的结果作为回调函数的参数。注意：``pg_callback()`` 也必须"锁定在内存中"。

在此非阻塞情况下，当收到填充完成的通知时，回调 ``pg_callback()`` 将执行以下操作：

-  验证 ``g_pftcb`` 非 NULL。
-  找到 ``g_pftcb`` 中等待填充完成的任务和 ``g_waitingforfill`` 列表头部任务之间较高的优先级。那将是等待填充的最高优先级任务的优先级。
-  如果此较高的优先级高于当前页面填充工作线程，则将工作线程的优先级提升到该级别。因此，页面填充工作线程将始终以等待填充的最高优先级任务的优先级运行。
-  保存填充操作的结果。
-  通知页面填充工作线程。

Task Resumption
^^^^^^^^^^^^^^^

对于非阻塞的 ``up_fillpage()``，当 ``g_pftcb`` 非 NULL 且从 ``pg_callback`` 收到填充完成状态时，页面填充工作线程将检测到页面填充已完成。在非阻塞情况下，当 ``up_fillpage()`` 返回时，页面填充工作线程将知道页面填充已完成。

在任何情况下，页面填充工作线程将：

-  验证状态信息和 ``g_pftcb`` 的一致性。
-  验证页面填充是否成功完成，如果是，则
-  调用 ``up_unblocktask(g_pftcb)`` 使刚接收填充的任务变为就绪状态。
-  检查 ``g_waitingforfill`` 列表是否为空。如果不是：

   -  从 ``g_waitingforfill`` 中移除等待页面填充的最高优先级任务，
   -  将任务的 TCB 保存在 ``g_pftcb`` 中，
   -  如果 ``g_pftcb`` 中线程的优先级高于页面填充工作线程的默认优先级，则将页面填充工作线程的优先级设置为该优先级。
   -  调用 ``pg_startfill()`` 启动下一次填充（如上所述）。

-  否则，

   -  将 ``g_pftcb`` 设置为 NULL。
   -  恢复页面填充工作线程的默认优先级。
   -  等待下一个填充相关事件（新的缺页）。

Architecture-Specific Support Requirements
------------------------------------------

Memory Organization
^^^^^^^^^^^^^^^^^^^

**内存区域**。芯片特定逻辑将虚拟和物理地址空间映射到三个通用区域：

#. 一个 .text 区域，包含"锁定在内存中"的代码，始终可用且永远不会导致缺页。此锁定内存在启动时加载并永久驻留。此内存区域必须包括：

   -  所有中断路径的所有逻辑。所有中断逻辑必须锁定在内存中，因为此处的设计不支持来自中断处理器的缺页。这包括从缺页处理器调用的缺页处理逻辑和 ``pg_miss()``。它还包括唤醒页面填充工作线程的 ``pg_callback()`` 函数以及调用 ``pg_callback()`` 的任何架构特定逻辑。
   -  IDLE 线程的所有逻辑。IDLE 线程必须始终准备好运行，不能因任何原因被阻塞。
   -  页面填充工作线程的所有部分必须锁定在内存中。此线程必须执行以解除任何等待填充的线程的阻塞。如果此线程被阻塞，将无法完成填充！

#. 一个 .text 区域，包含可以分配、映射到各种虚拟地址并从某些大容量存储介质填充的页面。
#. 以及用于 .bss、.text 和 .heap 的固定 RAM 空间。

此内存在下表中说明。注意：

-  虚拟地址空间中的页面与非易失性大容量存储设备中的 .text 页面之间存在一一对应的关系。
-  然而，可用的物理页面远少于虚拟页面。在任何给定时间，只有一部分物理页面会被映射到虚拟页面。此映射将根据程序执行的需要按需执行。

=============================  ============================  ====================
SRAM                           Virtual Address Space         Non-Volatile Storage
=============================  ============================  ====================
.                              DATA                          .
.                              Virtual Page *n* (*n* > *m*)  Stored Page *n*
.                              Virtual Page *n-1*            Stored Page *n-1*
DATA                           ...                           ...
Physical Page *m* (*m* < *n*)  ...                           ...
Physical Page *m-1*            ...                           ...
...                            ...                           ...
Physical Page *1*              Virtual Page *1*              Stored Page *1*
Locked Memory                  Locked Memory                 Memory Resident
=============================  ============================  ====================

**示例**。例如，假设 SRAM 大小为 192K（如 NXP LPC3131）。进一步假设：

-  锁定的、内存驻留的 .text 区域大小为 32K，以及
-  DATA 区域大小为 64K。
-  一个受管理页面的大小为 1K。
-  非易失性大容量存储设备上整个 .text 映像的大小为 1024K。

那么，锁定的内存驻留代码大小为 32K（*m*\\ =32 页）。物理页面区域大小为 96K（96 页），数据区域大小为 64 页。虚拟分页区域大小必须大于或等于 (1024-32) 或 992 页 (*n*)。

**构建锁定的内存映像**。实现此目的的一种方法是两阶段链接：

-  在第一阶段，创建一个部分链接的目标，包含所有中断/异常处理逻辑、页面填充工作线程加上 IDLE 线程的所有部分（必须始终可用于执行）。
-  此部分链接的所有 ``.text`` 和 ``.rodata`` 段应收集到单个段中。
-  第二次链接将链接部分链接的目标和剩余目标以产生最终二进制文件。链接器脚本应将"特殊"段放置在保留的、"不可交换"区域中。

Architecture-Specific Functions
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

大多数标准架构特定函数在 ``include/nuttx/arch.h`` 中声明。然而，对于此分页逻辑，架构特定函数在 ``include/nuttx/page.h`` 中声明。架构移植中应已提供的标准架构特定函数是 :c:func:`up_switch_context`。仅为按需分页支持而需要实现的新附加函数是：

.. c:function:: int up_checkmapping(FAR struct tcb_s *tcb)

  函数 ``up_checkmapping()`` 返回是否仍需要执行页面填充的指示。在某些情况下，缺页可能在多个线程上发生并被多次排队。此函数将防止同一页面被多次填充。

.. c:function:: int up_allocpage(FAR struct tcb_s *tcb, FAR void *vpage)

  此架构特定函数将在内存中预留页面并映射到其正确的虚拟地址。保存在 TCB 中的架构特定上下文信息将为函数提供识别虚拟缺失地址所需的信息。此函数将在 ``vpage`` 中返回已分配的物理页面地址。底层物理页面的大小由配置设置 ``CONFIG_PAGING_PAGESIZE`` 确定。注意：此函数必须 *始终* 返回页面分配。如果所有可用页面都在使用中（典型情况），此函数将选择一个正在使用的页面，取消映射，并使其可用。

.. c:function:: int up_fillpage(FAR struct tcb_s *tcb, FAR const void *vpage, void (*pg_callback)(FAR struct tcb_s *tcb, int result))

  使用来自非易失性存储的数据实际填充页面，必须通过单独调用架构特定函数 ``up_fillpage()`` 来执行。这将启动异步页面填充。公共分页逻辑将提供一个回调函数 ``pg_callback``，当页面填充完成（或发生错误）时将被调用。假定此回调在设备驱动完成填充操作时从中断级别发生。
